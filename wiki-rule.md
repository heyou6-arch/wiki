## 1. 객체 선택 정책

### 1-1. 정책 개요

**목적:** 드래그를 통해 블록을 전체 선택할 때, 선택된 모든 객체에 일관된 시각적 피드백을 제공하여 사용자가 편집 범위를 직관적으로 인지하게 합니다.

**배경:** 현재 이미지, 파일, 코드블럭, 토글 등 주요 컴포넌트들이 드래그 선택 시 시각적 변화가 없어 일괄 삭제나 이동 시 의도치 않은 데이터 손실이 발생하고 있습니다.

### 1-2. 객체 별 선택 시각화 정의

모든 기본 및 첨부 객체에 대해 '선택됨(Selected)' 상태를 정의합니다.

| 객체 분류 | 대상 컴포넌트 | 시각적 표기 방식 (Selected State) |
|---|---|---|
| 텍스트형 | 텍스트, 제목(H1/2/3), 리스트, 체크박스 | 시스템 기본 하이라이트 |
| 레이아웃형 | 표(Table), 인용, 토글 | 반투명 오버레이 + 외곽선 |
| 코드형 | 코드블럭 | 텍스트 시스템 기본 하이라이트 + 외곽선 |
| 첨부형 | 이미지, 파일 | 반투명 오버레이 + 외곽선 |
| 플로우 연동 | 플로우 업무, 플로우 게시글 | 반투명 오버레이 + 외곽선 |

### 1-3. 상세 인터랙션 가이드

**(1) 토글(Toggle) 선택 로직**

- **헤더 선택 시:** 사용자가 토글의 헤더(제목) 부분을 드래그에 포함할 경우, 접혀있거나 펼쳐진 내부 하위 콘텐츠 전체가 선택 영역으로 간주되어 함께 하이라이트 처리됩니다.
- **부분 선택:** 토글 내부의 특정 텍스트만 드래그할 경우 기존 텍스트 하이라이트 로직을 따릅니다.

## 2. 드래그 앤 드롭 통합 관리

### 2-1. 정책 개요

**목적:** 위키 내 모든 객체의 이동 수단을 '드래그 핸들'로 단일화하여 조작 일관성을 확보하고, 편집 중 발생하는 의도치 않은 레이아웃 수정을 방지합니다.

**배경:** 텍스트, 이미지뿐만 아니라 업무/게시글 등 상호작용이 필요한 카드형 객체에서 본체 드래그를 허용할 경우, 내부 콘텐츠 선택(텍스트 복사) 및 링크 클릭 동작과 간섭이 발생하여 사용자 경험이 저해될 수 있습니다.

### 2-2. 상세 정책

| 객체 유형 | 본체 드래그 | 핸들 드래그 | 비고 |
|---|---|---|---|
| 텍스트 / 제목 | 텍스트 선택 (기본 동작) | 블록 이동 | |
| 이미지 | 차단 | 블록 이동 | 본체 클릭 = 이미지 뷰어 or 선택 UI |
| 파일 | 차단 | 블록 이동 | 본체 클릭 = 파일 다운로드/미리보기 |
| 코드블럭 | 텍스트 선택 (내부 코드) | 블록 이동 | |
| 표(Table) | 셀 내 텍스트 선택 | 블록 이동 | 셀 내부 드래그 = 셀 텍스트 선택 |
| 인용 | 텍스트 선택 | 블록 이동 | |
| 토글 | 텍스트 선택 (내부) | 블록 이동 | 헤더 외 영역 = 내부 콘텐츠 선택 |
| 체크박스 | 텍스트 선택 | 블록 이동 | 체크 영역 클릭 = 체크 토글 |
| 플로우 업무 카드 | 차단 | 블록 이동 | 본체 클릭 = 업무 상세 열기 |
| 플로우 게시글 카드 | 차단 | 블록 이동 | 본체 클릭 = 게시글 상세 열기 |

**핵심 원칙:** 모든 블록 이동은 반드시 드래그 핸들을 통해서만 수행됩니다. 본체 드래그는 해당 객체 고유의 상호작용(텍스트 선택, 클릭 액션 등)에 사용됩니다.

## 3. 객체 인라인 삽입 정책

### 3-1. 정책 개요

**목적:** 위키 에디터 내 객체 삽입 시 허용되지 않는 중첩 구조로 인한 렌더링 오류와 데이터 손실을 방지하고, 사용자에게 예측 가능한 편집 환경을 제공합니다.

### 3-2. 기본 원칙

1. **진입점 통일:** / 메뉴 입력과 붙여넣기 두 진입점에 동일한 허용 규칙을 적용합니다.
2. **plain text 변환:** 허용되지 않는 객체가 붙여넣기될 경우 내용만 추출하여 plain text로 변환합니다.
3. **/ 메뉴 필터링:** 현재 커서가 위치한 컨테이너에서 허용되지 않는 객체는 / 메뉴에 노출하지 않습니다.
4. **코드 블록 예외:** 코드 블록 내부는 plain text 입력만 허용하므로 / 메뉴 자체를 비활성화합니다.

### 3-3. 전체 요약표

O = 허용, X = 차단

| 객체 | 표 | 인용 | 체크박스 | 순서목록 | 불릿목록 | 코드 | 토글 |
|---|---|---|---|---|---|---|---|
| 텍스트 | O | O | O | O | O | O | O |
| 제목 | X | O | X | X | X | X | O |
| 순서목록 | O | O | X | O | O | X | O |
| 불릿목록 | O | O | X | O | O | X | O |
| 체크박스 | O | X | O | X | X | X | O |
| 인용 | X | X | X | X | X | X | O |
| 코드 | O | O | X | X | X | X | O |
| 토글 | X | X | X | X | X | X | O |
| 이미지 | O | O | X | X | X | X | O |
| 파일 | O | O | X | X | X | X | O |
| 표 | X | O | X | X | X | X | O |
| 플로우 | X | X | X | X | X | X | O |

### 3-4. 붙여넣기 처리 정책

| 상황 | 처리 방식 |
|---|---|
| 허용 객체 붙여넣기 | 그대로 삽입 |
| 차단 객체 붙여넣기 | 내용만 추출 후 plain text로 변환 |
| 내용 없는 차단 객체 붙여넣기 | 빈 텍스트 블록으로 변환 |

### 3-5. 코드 붙여넣기 처리 정책

VSCode, Sublime 등 코드 에디터에서 복사한 내용을 붙여넣기 시 코드 블록 자동 변환 여부에 대한 정책.

| 소스 | 클립보드 포맷 | 위키 붙여넣기 결과 |
|---|---|---|
| VSCode / Sublime 등 코드 에디터 | text/plain only | 코드 블록 자동 변환 **없음** → 일반 텍스트로 출력 |
| 노션 코드 블록 복사 | 자체 포맷 | 코드 블록으로 변환 가능 (소스에 코드 블록 메타 포함 시) |

**현재 정책 (확정):**
- 코드 에디터(VSCode 등)에서 복사한 내용은 클립보드에 `text/plain`만 담기므로 위키가 코드 여부를 판단할 수 없음
- 코드 블록으로 자동 변환하지 않고 **일반 텍스트로 출력** — 정상 동작
- 사용자가 직접 `/코드` 슬래시 커맨드로 코드 블록 생성 후 붙여넣기 필요

> 노션의 경우 자체 포맷으로 코드 블록 메타가 포함될 수 있어 자동 변환이 가능할 수 있음. 추후 노션 연동 스펙 확정 시 별도 대응 검토.

## 4. 자동 서식 변환(Auto-formatting) 정책

### 4-1. 정책 개요

**목적:** 마크다운 스타일 단축 입력(숫자+마침표, 대시+스페이스 등)에 의한 자동 블록 변환 규칙을 블록 타입별로 정의하여, 제목 블록에서 번호 입력이 불가능한 문제 등 사용성 이슈를 해결합니다.

**배경 (QA 최다 빈출 이슈):** 제목(H1/H2/H3) 블록에서 '1. 개요'와 같이 번호를 포함한 제목을 입력하면 자동으로 순서 있는 목록으로 변환되어 제목 스타일이 해제됩니다.

### 4-2. 핵심 원칙

**자동 서식 변환은 '일반 텍스트' 블록에서만 동작합니다.** 제목, 인용, 코드, 테이블 셀 안에서는 모든 자동 변환이 비활성화됩니다.

### 4-3. 블록 타입별 자동 변환 분기

| 현재 블록 타입 | 숫자. + 스페이스 | - + 스페이스 | [] + 스페이스 |
|---|---|---|---|
| 일반 텍스트 | 순서 목록 변환 | 불릿 목록 변환 | 체크박스 변환 |
| 제목 (H1/H2/H3) | 변환 안 함 (텍스트 유지) | 변환 안 함 | 변환 안 함 |
| 인용 블록 | 변환 안 함 | 변환 안 함 | 변환 안 함 |
| 코드 블록 | 변환 안 함 | 변환 안 함 | 변환 안 함 |
| 테이블 셀 | 변환 안 함 | 변환 안 함 | 변환 안 함 |
| 토글 내부 텍스트 | 순서 목록 변환 | 불릿 목록 변환 | 체크박스 변환 |

### 4-4. 한글 IME 대응 정책

`compositionstart` ~ `compositionend` 이벤트 사이에는 모든 자동 변환을 무시합니다. 한글 조합 중 자동 변환이 트리거되면 입력 데이터가 손실될 수 있습니다.

### 4-5. 실행취소(Undo) 정책

| 항목 | 정책 |
|---|---|
| 자동 변환 후 Ctrl+Z | 변환 전 상태로 즉시 복원 (1단계 undo) |
| 토스트 알림 | 자동 변환 발생 시 "순서 목록으로 변환됨 · 실행취소" 토스트 1회 노출 |
| 토스트 자동 소멸 | 3초 후 자동 소멸, 클릭 시 즉시 실행취소 |

### 4-6. 들여쓰기 (Tab / Shift+Tab) 정책

**순서 목록 사용 시 텍스트 외 요소(이미지, 코드 블록 등)도 Tab 들여쓰기를 허용한다.**

| 객체 | Tab 들여쓰기 | Shift+Tab 내어쓰기 | 비고 |
|---|---|---|---|
| 텍스트 / 리스트 항목 | 허용 | 허용 | 기본 동작 |
| 이미지 | 허용 | 허용 | 블록 단위로 들여쓰기 |
| 파일 | 허용 | 허용 | 블록 단위로 들여쓰기 |
| 코드 블록 (블록 단위) | 허용 | 허용 | 블록 전체 들여쓰기. 코드 블록 내부 Tab은 코드 들여쓰기로 동작 |
| 표 | 허용 | 허용 | 표 전체 블록 단위 들여쓰기 |
| 플로우 카드 | 허용 | 허용 | 블록 단위로 들여쓰기 |

**Tab 키 동작 분기:**

| 상황 | Tab 동작 |
|---|---|
| 커서가 블록 외부 (블록 선택 상태) | 블록 들여쓰기 |
| 커서가 코드 블록 내부 | 코드 들여쓰기 (스페이스 2 또는 4칸) |
| 커서가 에디터 빈 영역 | 페이지 상단으로 포커스 이동 (기존 동작 유지) |
| 최대 들여쓰기 깊이 초과 | 더 이상 들여쓰기 안 됨 (최대 4단계) |

## 5. / 메뉴 컨텍스트 필터링 정책

### 5-1. 정책 개요

사용자가 / 를 입력했을 때 현재 커서가 위치한 컨테이너 블록에 따라 삽입 가능한 객체 목록만 노출합니다.

### 5-2. 컨테이너별 차단 객체

| 컨테이너 | / 입력 시 노출 차단 객체 |
|---|---|
| 표 셀 | 제목1/2/3, 인용, 토글, 표(중첩), 플로우 업무/게시글 |
| 인용 | 체크박스, 인용(중첩), 토글, 플로우 업무/게시글 |
| 체크박스 | 제목, 순서목록, 불릿목록, 인용, 코드, 토글, 이미지, 파일, 표, 플로우 |
| 순서목록 | 제목, 체크박스, 인용, 코드, 토글, 이미지, 파일, 표, 플로우 |

## 6. 문서 공개 범위 및 권한 정책

> v1.1 신규 추가

### 6-1. 정책 개요

**목적:** 문서 생성 시 공개 범위를 3단계(나만/공유 서재/회사 도서관)로 정의하고, 역할별 권한을 매핑하여 불필요한 문서 노출을 방지하고 공유 서재 단위 문서 관리를 가능하게 합니다.

**배경:** 기존에 "내 문서"와 "회사 도서관"만 구분되어 있어, 대부분의 문서가 회사 도서관 공개로 생성되고 사이드바가 무한 스크롤되는 문제가 발생합니다. "공유 서재" 레이어를 추가하여 관련 없는 문서가 보이지 않도록 합니다.

### 6-2. 문서 공개 범위 4단계

| 범위 | 볼 수 있는 사람 | 설명 | 기본값 |
|---|---|---|---|
| 나만 | 문서 작성자 본인 | 개인 메모, 초안 등. 다른 사람에게 보이지 않음. | - |
| **공유 서재** | **해당 공유 서재 멤버** | **초대된 멤버만 접근. 목적·조직 무관하게 자유 구성.** | **기본값** |
| 회사 도서관 | 워크스페이스 전 직원 | 사내 공지, 기준 문서 등. 모든 멤버가 조회 가능. | - |
| 링크가 있는 누구나 | 링크를 받은 모든 사용자 (외부인 포함) | 외부 공유 전용. 게스트 계정 생성 없이 링크만으로 접근. | - |

문서 생성 시 공개 범위 기본값은 **"공유 서재"**입니다.

**범위별 구분:**

| 구분 | 내부 공유 | 외부 공유 |
|---|---|---|
| 나만 / 공유 서재 / 회사 도서관 | O | X |
| 링크가 있는 누구나 | X | **O (외부 공개)** |

"링크가 있는 누구나"는 Flow 구성원이 아닌 외부인도 접근 가능한 상태이므로, 선택 시 **최고 관리자 승인이 필요**합니다 (섹션 6-8 참고).

### 6-3. "공유 서재"의 기준 정의

**"공유 서재" = 전 직원이 자유롭게 생성하고 초대로 참여하는 비공개 협업 공간**

조직도와 무관하게 목적에 따라 자유롭게 생성합니다. 기존 조직도 기반 자동 생성 방식은 폐지하며, 기존 팀 기반 문서는 회사 도서관으로 이관되고 구성원이 직접 재배치합니다.

**생성/참여 규칙:**

| 항목 | 정책 |
|---|---|
| 생성 권한 | 전 직원 (관리자 승인 불필요) |
| 참여 방식 | 초대만 (비공개). 공개 가입 없음. |
| 서재 수 제한 | 제한 없음 |
| 초대 권한 | 서재 소유자 + 편집자 이상 |

**기존 데이터 이관:**

| 항목 | 정책 |
|---|---|
| 기존 조직도 팀 자동 서재 | 폐지. 기존 문서는 회사 도서관으로 일괄 이관. |
| 이관 후 처리 | 구성원이 직접 공유 서재 생성 후 문서 재배치. |
| 이관 알림 | 기존 서재 소유자에게 "문서 N건이 회사 도서관으로 이관되었습니다" 알림 발송. |

**시나리오 예시:**

| 서재명 | 목적 | 구성원 |
|---|---|---|
| 위키 런칭 TFT | 프로젝트 단위 협업 | 기획 + 개발 + 디자인 혼합 |
| 상품기획팀 | 팀 단위 (자발적 재생성) | 상품기획팀원 |
| 2026 OKR 스터디 | 목적 기반 그룹 | 자유 참여 |

### 6-4. 권한 기본 원칙

> v4.0 업데이트 · 최상위 기준은 6-19 참조

**기본 원칙 (v4.0):**

| 권한 유형 | 기본 동작 |
|---|---|
| **조회** | 폴더 권한 **상속**이 기본 · 문서 개별 Override 가능 |
| **편집** | 문서 **개별 설정**이 기본 · 폴더 편집자는 Override 없는 문서만 제어 |

범위(나만/공유 서재/회사 도서관)는 "누가 볼 수 있는가"의 기본 경계를 결정합니다. 개별 문서에서는 Override를 통해 이 경계를 좁히거나 넓힐 수 있습니다 (단, 회사 도서관 조회는 Override 불가 — 6-19-8 참조).

| 범위 | 폴더 조회 (기본값) | 문서 편집 (기본값) |
|---|---|---|
| 나만 | 작성자만 | 작성자만 |
| 공유 서재 | 서재 멤버 전체 | 작성자만 (+ 수동 추가된 사람) |
| 회사 도서관 | 전 직원 (고정) | 작성자만 (+ 수동 추가된 사람) |

**Override 시나리오:**
- 공유 서재 문서 → 특정인에게만 조회 제한 (Override ON)
- 회사 도서관 문서 → 특정 팀에게만 편집 권한 (조회 Override는 불가)
- 모든 Override는 Manual Override Flag로 DB 기록 (6-19-6)

### 6-5. 수동 권한 추가 및 공유

소유자는 공개 범위와 무관하게 특정 인원에게 조회 또는 편집 권한을 개별 부여 (편집은 조회 포함)할 수 있습니다.

**권한 옵션 (2종):**

| 권한 | 설명 |
|---|---|
| 조회 | 읽기만 가능. 편집 불가. |
| 편집 | 읽기 + 편집 모두 가능. **조회 권한을 포함한다.** |

**플랜별 부여 가능 권한:**

| 플랜 | 대상 | 부여 가능 권한 |
|---|---|---|
| 무료 (멤버 10명 미만) | 누구든 | 조회 / 편집 |
| 유료 (멤버 6인 초과) | 유료 멤버 | 조회 / 편집 |
| 유료 (멤버 6인 초과) | 비유료 멤버 | 조회만 |
| 공통 | 미가입자 (외부인) | 링크 조회만 (별도 — 6-14 외부 공유 정책 따름) |

**수동 추가 대상:** 개별 사용자, 부서 단위 (부서 추가 시 해당 부서 전원에게 동일 권한 부여)

**2차 공유 (편집자의 권한 부여/회수):**

| 항목 | 정책 |
|---|---|
| 편집자 → 다른 사람에게 편집/조회 부여 | 가능 |
| 편집자 → 다른 사람의 권한 회수 | 가능 |
| 조회자 → 다른 사람에게 권한 부여 | 불가 |
| 조회자 → 다른 사람의 권한 회수 | 불가 |
| 2차 편집자의 3차 공유 | 동일하게 가능 (편집자 이상이면 부여/회수 모두 가능) |
| 공유 이력 추적 | "김원모가 박서윤에게 편집 부여" 로그 기록 |
| 소유자(작성자)의 회수 | 언제든 모든 수동 추가 권한 회수 가능 |

**핵심 원칙:** 편집자 이상 권한을 가진 사람은 해당 문서의 권한 부여/회수가 가능합니다. 조회자는 권한 관련 행위가 불가합니다.

### 6-5-1. 폴더 이동 시 조회/편집 분리 원칙

> v3.0 업데이트 · 문서 권한 우선 원칙 반영

문서를 다른 폴더로 이동할 때, **문서 권한은 유지**되고 폴더 조회 권한은 별도로 적용됩니다.

**핵심 원칙 (v3.0):**

| 구분 | 규칙 |
|---|---|
| 문서 조회 권한 | **기존 권한 유지** + 폴더 조회 권한자도 추가 노출 |
| 문서 편집 권한 | **기존 권한 유지** (폴더 이동에 영향 없음) |
| 폴더 조회 권한 | 폴더 기본 설정에 따름 (서재 멤버 자동 · 회사 도서관 전체 등) |

**즉, 문서 권한은 "지워지지 않고 누적"됩니다.** 더 넓은 쪽이 적용되는 원칙 (6-17 참고).

**이동 시나리오:**

| 케이스 | 문서 조회 (유지) | 폴더 조회 (추가 노출) | 문서 편집 (유지) |
|---|---|---|---|
| 공유 서재 문서를 회사 도서관으로 | 기존 조회자 유지 | + 회사 도서관 전 직원 | 작성자 + 기존 편집자만 |
| 회사 도서관 문서를 공유 서재로 | 기존 조회자 유지 | + 해당 서재 멤버 | 작성자 + 기존 편집자만 |
| 공유 서재 A → 공유 서재 B | 기존 조회자 유지 | + B 서재 멤버 · - A 서재 멤버 (폴더 접근 상실) | 작성자 + 기존 편집자만 |
| 공유 서재 → 나만 | 기존 조회자 유지 (수동 부여된 사람 포함) | - 서재 멤버 (폴더 접근 상실) | 기존 편집자 유지 |

**중요 변경점 (v2.9 → v3.0):**

| v2.9 이전 | v3.0 |
|---|---|
| 조회 권한이 "폴더를 따름"으로 재설정 | 조회 권한 **유지 + 폴더 권한 추가** |
| 이동 시 기존 수동 부여 조회자 상실 가능 | 기존 수동 부여 조회자 **그대로 유지** |

이 변경으로 "이동했더니 권한이 사라졌다" 이슈를 방지합니다.

### 6-5-2. 퇴사 시 문서 소유권 정책

작성자 = 소유자로 일원화하며, 퇴사(계정 비활성화) 시 아래 규칙을 따릅니다.

| 문서 유형 | 퇴사 시 처리 |
|---|---|
| 공유 서재 내 문서 | 소유자(작성자)가 최고 관리자로 자동 이전 |
| 회사 도서관 폴더 내 문서 | 소유자(작성자)가 최고 관리자로 자동 이전 |
| 나만 문서 | 본인 유지 (계정 비활성화 상태로 보관) |
| 최고 관리자 이관 후 | 최고 관리자가 새 소유자를 지정하거나 직접 관리 |

**퇴사 시 알림:** 소유권이 이전되면 최고 관리자에게 "지수경(퇴사)의 문서 N건이 이관되었습니다. 새 소유자를 지정해주세요." 알림을 발송합니다.

### 6-6. 권한 우선순위

> v4.0 업데이트 · 최상위 기준은 6-19 참조

권한 충돌 시 아래 우선순위로 판단합니다:

| 우선순위 | 계층 | 설명 |
|---|---|---|
| 1 (최우선) | **Override된 권한** (Manual Override Flag ON) | 개별 문서에 명시적으로 설정된 권한. 폴더 권한 변경에도 유지됨. |
| 2 | **문서 편집 권한 (Override)** | 편집은 문서 개별 설정이 기본값 |
| 3 | **폴더 조회 상속 권한** | 조회는 폴더 상속이 기본. Override 없으면 이것이 적용 |
| 4 (기본값) | 레이어 기본값 | 내 문서=본인만, 공유 서재=서재 멤버, 회사 도서관=전 직원 |

**핵심 우선순위 규칙:**

- **Override는 항상 우선:** 폴더가 변경되어도 Override 문서는 영향받지 않음 (Manual Override Flag)
- **Override = 개별 권한 보호 영역:** Override된 문서는 **폴더 편집자도 제어 불가** (6-19-7)
- **조회는 폴더 상속, 편집은 문서 우선:** 조회는 폴더 기본값 + 예외 Override / 편집은 문서 개별 설정이 일반
- **회사 도서관 조회 예외:** 조회 Override 불가 (전 직원 공개 원칙 유지)
- **범위 축소 vs 확대:** 범위 좁히기 자유, 넓히기는 경고 팝업 표시 (6-16)

### 6-7. 충돌 시나리오

| # | 상황 | 결과 |
|---|---|---|
| 1 | 폴더 "회사 도서관 조회" + 수동으로 지수경에게 "편집" 부여 | 전 직원 조회, 지수경만 편집 (수동 우선) |
| 2 | 폴더 "공유 서재 조회" + 문서에서 "공유 서재 조회" 유지 + 수동 추가 없음 | 서재 멤버 조회, 작성자만 편집 (기본 원칙) |
| 3 | 폴더 "공유 서재" + 문서를 "회사 도서관"로 확대 시도 | 경고 팝업 표시 후 확인 시 적용 |
| 4 | "나만" 문서 + 수동으로 김원모에게 "조회" 부여 | 작성자 편집, 김원모 조회, 나머지 접근 불가 |
| 5 | 상위 폴더 "회사 도서관 조회" + 하위 폴더 "공유 서재 조회+편집" | 하위 폴더 내 문서는 공유 서재 조회+편집 권한 적용 (하위 폴더 권한이 상위를 덮어씀) |
| 6 | 문서 "공유 서재 조회" + 수동으로 박서윤(서재 외부 멤버)에게 "편집" 부여 | 서재 멤버 조회, 박서윤 편집, 나머지 접근 불가 (수동 추가는 범위와 독립) |
| 7 | 유료 플랜 + 비유료 멤버에게 "편집" 부여 시도 | 차단 — 비유료 멤버에게는 조회만 부여 가능. 팝업에서 "편집" 옵션 비활성. |

| 역할 | 정의 | 사이드바 |
|---|---|---|
| 최고 관리자 | 워크스페이스 전체 어드민. 플랜/멤버/외부 공유 전체 설정 가능. | 노출 |

- **전 직원 조회 고정**: 일반 접근 권한 영역에 "조회 (고정)" 라벨 표시 · 드롭다운 없음 · Override 불가
- **편집자 설정**: 검색창으로 내부 멤버 검색 → 편집자 리스트에 즉시 추가 (권한: 조회+편집 고정)

- **저장**: [저장] 클릭 시 편집자 리스트 · 외부인 조회 권한 DB 반영

> **외부인 초대는 6-14 외부 공유 승인 플로우 준수**

#### 6-16-3. 📄 내 문서 공유 팝업

| 1 | **문서 권한이 폴더 권한을 override** · 문서 권한이 더 넓으면 문서 권한 적용 |
| 2 | 폴더 구조상 **권한 없는 문서**는 **제목 노출 · 선택 불가 · 🔒 표시** |
| 3 | 폴더 조회 권한자는 해당 폴더에 **새 문서 생성 가능** |
| 4 | 생성된 문서의 권한은 **문서 레벨에서 별도 설정** · 폴더 권한과 독립 |

#### 6-17-2. 📚 공유 서재 폴더 권한

> v4.0 업데이트 · 조회+편집 둘 다 설정 가능

**기본 원칙:** 서재 멤버는 서재 내 모든 폴더에 **자동 조회 권한** 부여. 폴더별로 편집 권한도 개별 설정 가능.

| 항목 | 정책 |
|---|---|
| 조회 권한 기본값 | 서재 멤버 전원 자동 부여 |
| 조회 권한 제한 | 특정 폴더를 **일부 멤버에게만 제한** 가능 |
| 편집 권한 | **개별 설정 가능** (서재 멤버 중 일부 또는 전원) |
| 조회/편집 설정 권한자 | **서재 소유자 + 서재 관리자** |
| 비멤버 접근 | 서재 외부인은 폴더 조회 불가 |

**폴더 편집 권한자가 할 수 있는 일:**

- 폴더 내 **새 문서·하위 폴더 생성**
- 기존 문서의 이동·이름 변경·삭제 (**단, Override 문서 제외 — 6-19-7**)
- 폴더 공유 설정 변경

**시나리오 예시:**

| 상황 | 결과 |
|---|---|
| 서재 멤버 5명 · 새 폴더 생성 | 5명 자동 조회 · 편집은 생성자만 (기본) |
| 팀 리드에게 폴더 편집 권한 부여 | 리드가 폴더 내 문서 관리 가능 |
| 특정 문서에 Override 설정 | 폴더 편집자도 그 문서는 건드릴 수 없음 |
| 폴더 조회 없는 멤버 · 문서 조회 권한 있음 | 문서 직접 접근 가능 (링크로) |

#### 6-17-3. 🏢 회사 도서관 폴더 권한

**기본 원칙:** 조회는 전 직원 고정이고, **편집 권한만 폴더 레벨에서 설정**합니다.

| 항목 | 정책 |
|---|---|
| 조회 권한 | **전 직원 고정** · 개별 조정 불가 |
| 편집 권한 | 폴더 레벨에서 **지정한 팀/역할만** 허용 |
| 설정 권한자 | **최고 관리자 + 해당 폴더 소유자** |
| 편집 권한자 기본값 | **폴더 소유자만** (초기) |
| 편집 권한 확장 | 팀 단위 · 개인 단위 모두 가능 |

**시나리오 예시:**

| 폴더 | 조회 | 편집 |
|---|---|---|
| 📁 회사 공지사항 | 전 직원 | 경영팀만 |
| 📁 HR 정책 | 전 직원 | HR팀만 |
| 📁 기술 블로그 | 전 직원 | 개발팀 + 선정된 기고자 |
| 📁 보안 가이드 | 전 직원 | 보안 담당자 2명만 |

**왜 회사 도서관만 편집 권한을 폴더 레벨에서 설정하는가:**

회사 도서관은 **공식 문서 · 기준 문서 · 정책 문서** 중심으로 "누가 수정할 수 있는가"가 중요합니다. 모든 직원이 편집할 수 있다면 공식 문서 신뢰성이 훼손되므로, 폴더 단위로 편집 권한을 제한합니다.

반면 공유 서재는 "같이 일하는 TF/팀 공간"이므로 편집 권한은 문서 단위 유연성이 필요하고, 폴더 편집 권한 개념이 불필요합니다.

#### 6-17-4. 📄 내 문서 폴더 권한

**기본 원칙:** 내 문서는 **본인 전용 영역**. 폴더 권한 개념 자체가 없음.

| 항목 | 정책 |
|---|---|
| 폴더 권한 | **설정 불가** (폴더 자체가 본인 전용) |
| 외부 공유 | **문서 단위**에서만 가능 (문서를 타인에게 공유) |
| 폴더 내 문서를 타인 공유 | 각 문서 개별 공유 설정 |
| 공유받은 사람이 보는 경로 | 해당 문서만 노출 · 내 문서 폴더 구조 비노출 |

**폴더 전체를 타인에게 공유하고 싶다면:**
- 공유 서재를 생성하여 해당 폴더 내용을 이관
- 또는 각 문서를 개별 공유

#### 6-17-5. 권한 조합 매트릭스 (전 레이어)

| 폴더 조회 | 폴더 편집 | 문서 권한 | 실제 적용 |
|---|---|---|---|
| 있음 | - | 없음 | 폴더 안에 **표시되지 않음** (완전 숨김 · v3.2) |
| 있음 | - | 조회 | 문서 **조회 가능** |
| 있음 | 있음 (회사 도서관만) | 조회 | 문서 조회 가능 · **폴더 메뉴 편집 가능** (새 문서 생성 등) |
| 있음 | 있음 (회사 도서관만) | 편집 | 문서 편집 가능 |
| 없음 | - | 조회 | 문서 링크로 조회 가능 · 폴더엔 미노출 |
| 없음 | - | 편집 | 문서 링크로 편집 가능 · 폴더엔 미노출 |
| 없음 | - | 없음 | 접근 불가 · 링크로 접근 시 **"권한 없음" 안내** |

#### 6-17-6. 권한 없는 문서 UI 표시 (전 레이어 공통)

> v3.2 변경 · "제목 노출" → "아예 숨김"

폴더 안에 접근 권한 없는 문서는 **UI에서 완전히 숨김** 처리합니다.

| 항목 | 처리 |
|---|---|
| 폴더 목록 | 권한 없는 문서는 **표시되지 않음** |
| 문서 개수 카운트 | 본인이 접근 가능한 문서만 카운트 |
| 검색 결과 | 권한 없는 문서는 결과에서 **완전 제외** |
| 직접 링크 접근 | "이 문서에 접근 권한이 없어요" 안내 + [권한 요청] 버튼 |

**변경 이유:**
- v3.0 "제목 노출" 방식은 **보안 우려** (문서명만으로도 민감 정보 유출 가능 · 예: "2026년 구조조정 계획")
- 권한 없는 문서의 존재 자체를 숨겨 **정보 보안 강화**
- UI 단순성 증가 (🔒 아이콘 · 회색 처리 등 복잡 요소 제거)

**다른 사람이 공유한 링크로 접근 시:**

링크로 직접 진입할 때는 "해당 문서가 존재함"을 알 수 있는 상태이므로, 이 경우만 **"접근 권한 없음" 안내 + [권한 요청]** 을 노출합니다.

#### 6-17-7. 인용 허브 콘셉 폐기

v2.9까지 논의된 "인용 허브" 개념은 폴더 조회 권한 도입으로 우회 방식이 불필요해져 **전면 폐기**합니다.

| 인용 허브가 풀려던 것 | 대체 방식 |
|---|---|
| TF 자료를 한곳에 모으기 | 공유 서재 내 폴더 + 폴더 조회 권한 |
| 여러 관점에서 자료 탐색 | 공유 서재 멤버십 + 폴더 구조 |
| 권한 없이 TF 협업 공간 구성 | 서재 멤버십 + 폴더 조회 권한 + 문서 권한 |
| 카드 배너 시각화 | 필요시 **Phase 2에서 폴더 뷰 개선** 검토 (범위 별도) |

### 6-18. 폴더 권한 엣지케이스 정책

> v3.2 신규 추가 · 8개 논점 결정 사항 반영

#### 6-18-1. 문서 이동 시 권한 일괄 정리

문서가 여러 폴더를 거쳐 이동하면서 조회 권한이 무한 누적되는 것을 방지합니다.

**일괄 정리 팝업 플로우:**

| 단계 | 동작 |
|---|---|
| 1. 이동 시도 | 대상 폴더 선택 후 이동 버튼 클릭 |
| 2. 권한 정리 팝업 | "현재 권한 N명" 리스트 노출 + 정리 옵션 |
| 3. 정리 옵션 | **[기존 권한 유지] / [자동 부여만 회수] / [모두 회수] / [개별 선택]** |
| 4. 확인 | 선택한 방식으로 권한 재설정 + 이동 실행 |

**기본 권장 옵션:** "자동 부여만 회수" (서재 멤버십으로 자동 부여된 권한은 회수 · 수동 부여는 유지)

**권한 리스트 표시:**

| 구분 | 표시 |
|---|---|
| 자동 부여 | 🤖 "서재 멤버십으로 부여됨" 배지 |
| 수동 부여 | 👤 "직접 공유됨" 배지 |

#### 6-18-2. 공유 서재 → 회사 도서관 이동 보안

공유 서재 폴더를 회사 도서관으로 이동하면 폴더 내 모든 문서가 **갑자기 전 직원에게 노출**될 수 있습니다. 이를 방지하기 위해 강화된 확인 플로우를 적용합니다.

**이동 확인 팝업:**

| 요소 | 내용 |
|---|---|
| 헤더 | **⚠️ 전 직원 공개 주의** (빨간 배경) |
| 경고 메시지 | "이 폴더 안의 **{N}개 문서가 전 직원에게 공개**됩니다. 공개해도 되는지 확인해주세요." |
| 문서 리스트 | 영향받는 문서 목록 최대 10개 표시 + "외 N개" |
| 민감 정보 경고 | 문서 태그에 "#기밀", "#비공개", "#임원" 등이 있으면 추가 경고 |
| 확인 체크박스 | **"위 문서들이 전 직원에게 공개됨을 확인했습니다"** (필수 체크) |
| 최종 확인 | 체크박스 체크 후에만 [이동하기] 버튼 활성화 |
| 취소 옵션 | [취소] / [개별 문서 권한 유지한 채 이동] |

**"개별 권한 유지한 채 이동" 옵션:**
- 폴더만 회사 도서관으로 이동
- 폴더 내 문서들의 조회 권한은 **기존 서재 멤버로 유지**
- 회사 도서관 폴더지만 안의 문서는 기존 공유 범위 유지 (폴더 권한보다 문서 권한 우선 원칙)

#### 6-18-2-1. Override 문서의 회사 도서관 이동 시 충돌 처리

공유 서재에서 조회 Override(범위 축소)가 설정된 문서를 회사 도서관으로 이동할 경우 6-19-8과 충돌이 발생합니다.

| 상황 | 처리 방식 |
|---|---|
| **조회 Override된 문서** → 회사 도서관 이동 | **이동 차단** · "회사 도서관은 전 직원 조회가 고정입니다. 개별 조회 제한을 해제한 후 이동해주세요" |
| Override 해제 후 이동 | 정상 이동 가능 · 이동 후 전 직원 조회 자동 적용 |
| **편집 Override된 문서** → 회사 도서관 이동 | 이동 허용 · 이동 후에도 편집 Override 유지 |

> **원칙 우선순위:** 6-19-8 "회사 도서관 조회 Override 불가" > 6-18-2 이동 허용 옵션


#### 6-18-2-2. 회사 도서관 → 내 문서 이동 차단

회사 도서관은 전 직원 공개 공간으로, **내 문서(개인 공간)로의 이동은 차단**됩니다.

| 이동 방향 | 처리 |
|---|---|
| 회사 도서관 → 내 문서 | **차단** · "회사 도서관 문서는 개인 문서로 이동할 수 없습니다" |
| 회사 도서관 → 공유 서재 | 허용 · 이동 후 서재 멤버 기준 접근 (전 직원 공개 해제) |
| 내 문서 → 회사 도서관 | 허용 · 이동 후 전 직원 조회 자동 적용 |

> **근거:** 회사 도서관은 조직 공식 문서 공간. 개인화(privatization)는 의도치 않은 정보 유실 위험.


#### 6-18-3. 회사 도서관 편집 권한 UI

회사 도서관 폴더 권한 설정 화면은 **편집자만 노출**하고, 조회는 안내문으로 처리합니다.

**권한 설정 팝업 구성:**

| 영역 | 내용 |
|---|---|
| 조회 권한 안내 | 상단에 고정 안내 박스: "📌 회사 도서관 폴더는 **전 직원이 자동 조회** 가능합니다" |
| 편집자 섹션 | "편집 권한" 헤더 + 멤버 리스트 + [멤버 추가] 버튼 |
| 편집자 기본값 | 폴더 소유자만 (초기) |
| 편집자 추가 | 개인 · 팀 단위 모두 가능 |
| 편집자 제거 | 소유자는 제거 불가 (소유자 이관 기능으로만 변경) |

**왜 조회는 안내문만:**
- 모든 회사 도서관 폴더의 조회 권한은 "전 직원"으로 고정
- 변경 불가능한 속성을 UI에 두면 혼란 야기
- 안내문으로 명확히 전달 + 편집자 설정에만 집중

#### 6-18-4. 사이드바 공유받은 문서 폴더

공유 서재 섹션 안에 **"📬 공유받은 문서"** 고정 폴더를 두어 개별 공유받은 문서들을 모아둡니다.

**사이드바 구조:**

```
📚 공유 서재
  ├ 📬 공유받은 문서 (고정)
  │    ├ 📄 개별 공유받은 문서들
  │    └ ...
  ├ 📁 기획팀 서재 (멤버인 경우만)
  └ 📁 브랜드 리뉴얼 TF
```

**"공유받은 문서" 폴더 특성:**

| 항목 | 정책 |
|---|---|
| 위치 | 공유 서재 섹션 최상단 고정 |
| 삭제 불가 | 사용자가 삭제할 수 없는 시스템 폴더 |
| 자동 수집 | 개별 공유받은 문서 자동 추가 |
| 정렬 | 공유받은 날짜 순 (최신 상단) |
| 비표시 조건 | 공유받은 문서가 0개면 폴더 자체 숨김 |

**공유 서재 멤버 아닌 경우:**
- 공유 서재 섹션 자체는 노출 (공유받은 문서 폴더만 보임)
- 실제 서재 하위 폴더는 없음

#### 6-18-5. 멤버 이탈 시 권한 일괄 회수

**모든 이탈 시나리오**에서 해당 사용자의 권한을 **일괄 회수**합니다.

| 이탈 유형 | 처리 |
|---|---|
| ① 본인 나가기 | 해당 서재 관련 모든 권한 회수 (자동 + 수동 모두) |
| ② 소유자가 내보냄 | 해당 서재 관련 모든 권한 회수 (자동 + 수동 모두) |
| ③ 회사 퇴사 | 전 워크스페이스 모든 권한 회수 + 기존 6-5-2 정책 적용 |
| ④ 서재 해체 | 해당 서재 관련 모든 멤버 권한 해체 · 문서는 소유자에게 반환 |

**정책 단순화 이유:**
- "수동 부여는 유지" 같은 예외 처리가 버그 유발 가능성 높음
- **협업 재개가 필요하면 수동 재부여** 가 더 명확
- 감사 관점에서 권한 이력 추적 용이


**Override 문서 처리 (A-3):**

| 상황 | 처리 방식 |
|---|---|
| 이탈 멤버가 편집 권한 있던 Override 문서 | 편집 권한 회수 · **Override Flag 유지** (소유자 권한 유지) |
| 이탈 멤버가 소유자인 Override 문서 | 소유권 최고 관리자로 자동 이전 · Override Flag 유지 |
| 이탈 멤버가 유일한 편집자인 Override 문서 | 최고 관리자에게 "편집자 없는 Override 문서 N개 확인 필요" 알림 |

> **원칙:** 멤버 이탈은 해당 멤버의 권한만 회수하며 Override 설정 자체는 변경하지 않습니다.
#### 6-18-5-1. 공유 서재 해체 시 문서 처리

공유 서재가 해체(삭제)될 경우 서재 내 문서는 아래와 같이 처리됩니다.

| 상황 | 처리 방식 |
|---|---|
| 서재 내 문서 (일반) | 서재 소유자의 **내 문서**로 자동 이동 |
| Override 설정된 문서 | 내 문서로 이동 후 Override Flag 유지 |
| 서재 소유자가 이미 탈퇴한 경우 | 최고 관리자의 내 문서로 이동 + 관리자 알림 발송 |
| 공유받은 외부 멤버의 접근 | 서재 해체 즉시 접근 차단 · "이 공유 서재가 해체되었습니다" 안내 |

> **서재 해체 vs 멤버 이탈(6-18-5) 차이:** 이탈은 멤버 개인 권한 회수. 해체는 서재 자체가 사라지므로 문서를 안전한 공간(소유자 내 문서)으로 이동.

#### 6-18-6. 회사 도서관 폴더 생성 권한

| 폴더 레벨 | 생성 권한자 |
|---|---|
| **최상단 폴더** (회사 도서관 바로 아래) | **최고 관리자 + 관리자만** |
| 하위 폴더 (최상단 폴더의 자식) | **전 직원** (폴더 조회 권한자) |
| 하위 하위 폴더 | **전 직원** (해당 폴더 조회 권한자) |

**설계 의도:**
- 최상단 폴더 = **회사의 공식 분류 체계** (예: "인사", "재무", "기술")
- 하위 폴더 = **팀/부서의 자율적 구조화**
- 최상단만 통제하여 **일관성 + 유연성** 동시 확보

#### 6-18-7. 신규 문서 생성 시 기본 권한

폴더 조회 권한자가 새 문서를 만들면 다음과 같이 기본 권한이 부여됩니다.

**공유 서재 내 새 문서:**

| 권한 | 기본값 |
|---|---|
| 조회 | 해당 폴더 조회 권한자 전원 자동 부여 |
| 편집 | 작성자만 |
| 변경 가능 | 작성자가 문서 공유 설정에서 조정 |

**회사 도서관 내 새 문서:**

| 권한 | 기본값 |
|---|---|
| 조회 | 전 직원 자동 (회사 도서관 원칙) |
| 편집 | 소유자 + 폴더 편집 권한자 전원 |

**내 문서 내 새 문서:**

| 권한 | 기본값 |
|---|---|
| 조회 | 작성자만 |
| 편집 | 작성자만 |

#### 6-18-8. 폴더 삭제 정책

**폴더는 비어있을 때만 삭제 가능**합니다.

| 상황 | 동작 |
|---|---|
| 빈 폴더 | 삭제 가능 · 확인 팝업 1회 |
| 문서 있음 | **삭제 불가** · "폴더 안에 문서가 있어요. 먼저 이동하거나 삭제해주세요" |
| 하위 폴더 있음 | 삭제 불가 · 동일 안내 |
| 하위 폴더 안 문서 있음 | 삭제 불가 · "N개 문서가 포함되어 있어요" |

**삭제 불가 시 빠른 액션:**

| 버튼 | 동작 |
|---|---|
| [일괄 이동] | 폴더 안 모든 문서를 다른 폴더로 한번에 이동 (위치 선택 팝업) |
| [일괄 삭제] | 폴더 안 모든 문서 일괄 삭제 (신중하게 · 빨간 경고) |
| [취소] | 삭제 중단 |

**권한 없는 문서가 있는 경우:**
- 삭제 권한자가 접근 못하는 문서가 폴더 안에 있으면 **삭제 차단**
- "접근 권한 없는 문서 N개가 있어 삭제할 수 없습니다" 안내
- 해당 문서 소유자에게 확인 요청하거나 이동 요청 필요


**데드락 방지 — 접근 권한 없는 문서가 폴더 안에 있을 경우:**

| 상황 | 처리 방식 |
|---|---|
| 폴더 삭제 시도 + 내부에 **권한 없는 문서 존재** | 삭제 차단 · "이 폴더에 접근 권한이 없는 문서 N개가 있어요. 해당 문서를 이동하거나 소유자에게 요청하세요" |
| 최고 관리자의 폴더 삭제 시도 | 권한 없는 문서 포함 시에도 **강제 삭제 가능** (어드민 페이지 > 페이지 관리에서 처리 후 삭제 권장) |
| 권한 없는 문서 이동 불가 + 삭제 불가 → 데드락 | 데드락 발생 시 자동 감지 → "소유자에게 폴더 삭제 요청" 메시지 + **최고 관리자에게 알림** 발송 |

> **데드락 정의:** 삭제도 불가 · 이동도 불가 · 목록도 안 보이는 상태. 최고 관리자 개입으로만 해소 가능.

### 6-19. 권한 체계 최종 모델 (v4.0)

> v4.0 신규 · 팀장님 협업 툴 권한 체계 설계 가이드 V2 기반 · **이 섹션이 모든 권한 충돌의 최상위 기준**

본 섹션은 Flow Wiki의 권한 체계를 상속 + Override 모델로 최종 정의합니다. 기존 섹션 (6-4, 6-5-1, 6-6, 6-17, 6-18)에서 정의된 세부 정책은 본 섹션의 원칙에 따라 해석됩니다.

#### 6-19-1. 최상위 원칙 (Top-Level Principles)

| # | 원칙 | 설명 |
|---|---|---|
| 1 | **조회는 폴더 상속** | 기본적으로 문서는 상위 폴더의 조회 권한을 상속받습니다 |
| 2 | **편집은 문서 개별 설정** | 편집 권한은 문서마다 개별 지정이 기본입니다 |
| 3 | **Override 가능** | 개별 문서는 폴더 상속과 다른 권한을 설정할 수 있습니다 |
| 4 | **Override = 개별 권한 보호 영역** | 문서에 Override가 설정되면 폴더 편집자도 해당 문서를 제어할 수 없습니다 |
| 5 | **Manual Override Flag** | DB에 Flag를 기록하여 폴더 권한 변경 시에도 개별 설정을 유지합니다 |
| 6 | **회사 도서관 조회는 예외** | 전 직원 공개 원칙 유지를 위해 조회 Override 불가 (편집 Override는 가능) |

#### 6-19-2. 권한 대상 (Who) 4종

팀장님 가이드 기반 + Flow Wiki 확장:

| 대상 | 정의 | Flow Wiki 매핑 |
|---|---|---|
| **소유자 (Owner)** | 데이터 완전 제어권 보유자 (생성자 또는 승계자) | 작성자 + 퇴사 시 이관자 |
| **조직 구성원 (Organization)** | 워크스페이스 내 모든 사용자 | 회사 도서관 조회 대상 (전 직원) |
| **특정 사용자 (Individual)** | 개별 부여받은 개인/그룹 | 공유 서재 멤버, 수동 부여자, 부서 부여 |
| **링크 사용자 (Link Access)** | URL을 보유한 모든 사용자 (익명 포함) | "링크가 있는 누구나" (6-14) |

#### 6-19-3. 폴더 권한 (Folder Permissions)

**편집자 (Editor):**

| 가능 행위 | 설명 |
|---|---|
| 폴더 내부 파일·하위 폴더 생성 | 새 문서·폴더 추가 |
| 기존 파일의 수정·이동·이름 변경·삭제 | **단, Override 설정된 문서는 예외 (6-19-7 참조)** |
| 폴더 전체의 공유 설정 변경 | 폴더 권한 부여/회수 |
| 다른 사용자 초대 | 폴더 조회/편집 권한 부여 |

**뷰어 (Viewer):**

| 가능 행위 | 설명 |
|---|---|
| 폴더 구조 확인 | 폴더 계층 탐색 |
| 내부 파일 목록 조회 | 파일명 · 메타 정보 열람 |
| 내부 파일 내용 열람 | 파일 자체에 별도 제한 없는 경우에만 |
| 이동·삭제·수정 | **불가** |

#### 6-19-4. 문서 권한 (File Permissions)

**편집자 (Editor):**

| 가능 행위 | 설명 |
|---|---|
| 본문 내용 수정·실시간 협업 | 에디터 편집 |
| 파일 이름 변경·버전 관리·사본 만들기 | 메타 편집 |
| 다른 사용자에게 공유·권한 부여 | 2차 공유 가능 |

**뷰어 (Viewer):**

| 가능 행위 | 설명 |
|---|---|
| 파일 내용 읽기·다운로드 | 설정에 따라 복사/인쇄 제한 가능 |
| 내용 수정·공유 설정 변경 | **불가** |

#### 6-19-5. 상속 규칙 (Inheritance)

**기본 상속 방향:**

```
폴더 권한 ─→ 문서 권한 (기본 상속)
```

| 시나리오 | 기본 동작 |
|---|---|
| 폴더에 문서 생성 | 폴더 권한이 문서에 상속됨 |
| 폴더 권한 변경 (예: 조회자 추가) | Override 없는 모든 하위 문서에 자동 반영 |
| 중첩 폴더 (상위 → 하위 → 문서) | 최하위 부모 폴더 권한 상속 (근접 우선) |
| 문서를 다른 폴더로 이동 | 이동 대상 폴더 권한 상속 (단, Override는 유지 — 6-19-6) |

**조회 vs 편집의 상속 차이:**

| 권한 유형 | 기본 적용 방식 |
|---|---|
| **조회** | 폴더 권한이 **기본값** · 문서 개별 설정은 **예외 (Override)** |
| **편집** | 문서 개별 설정이 **일반적** · 폴더 편집자는 폴더 관리 권한만 (Override 없는 한도 내) |

#### 6-19-6. Override 설정 (Manual Override Flag)

**Override란:**

폴더 상속을 따르지 않고 특정 문서에 **개별 권한을 명시적으로 설정**하는 것. DB에 `Manual Override Flag = true`로 기록됩니다.

**Override 발생 조건:**

| 행위 | Flag 변화 |
|---|---|
| 공유 팝업에서 권한 수정/멤버 추가·삭제 후 **[저장] 버튼 클릭** | Flag → **true** (ON) · DB 기록 확정 |
| [저장] 전 변경사항은 임시 상태 (닫기 시 파기) | Flag 변화 없음 |
| 문서 권한 설정을 해제 후 [저장] | Flag → **false** (OFF) · 폴더 권한 재상속 |
| 문서 권한은 그대로 두고 폴더 권한만 변경 | Flag 변화 없음 · 기존 값 유지 |

> **핵심 원칙: 편집 행위 ≠ Override 확정. [저장] 버튼 클릭 = Override 확정의 유일한 트리거.**
> 저장 전까지는 임시 변경 상태이며, 팝업을 닫으면 변경사항이 파기됩니다.

**Override된 문서의 동작:**

| 상황 | 결과 |
|---|---|
| 폴더 권한 변경 | Override 문서는 **영향받지 않음** (개별 권한 유지) |
| 문서를 다른 폴더로 이동 | Override 유지 (개별 권한 그대로) |
| Override 해제 요청 | 폴더 권한으로 복원 · Flag OFF |

**소유자 이전 시 Override 승계:**

| 상황 | Override 처리 |
|---|---|
| 소유자가 타인에게 소유권 이전 | Override Flag **그대로 유지** · 새 소유자가 Override 권한 승계 |
| 최고관리자에 의한 강제 이전 | Override Flag 유지 · 감사 로그 기록 |
| 원 소유자의 문서 접근 | 이전 후 일반 멤버 권한으로 접근 · Override 해제 권한 없음 |

> **원칙:** Override Flag는 문서에 귀속되며 소유자 변경과 무관하게 유지됩니다.

**Override 상태 UI 표시:**

| 위치 | 표시 |
|---|---|
| 문서 목록 | 🔒 아이콘 (Override 표시) |
| 문서 상세 | "이 문서는 개별 권한이 설정되어 있어요" 안내 |
| 폴더 설정 | "Override 문서 N개" 가시화 |


**검색 결과에서의 Override 문서 노출:**

| 검색 주체 | Override 문서 노출 여부 |
|---|---|
| Override 문서의 소유자 | 정상 노출 |
| Override 문서의 편집 권한자 | 정상 노출 |
| Override 문서의 조회 권한자 | 정상 노출 |
| **위 권한 없는 사람** | **비노출** (존재 자체 숨김 · 6-18-8 정책 동일 원칙 적용) |

> **원칙:** Override 자체가 검색 노출에 영향을 주지 않습니다. 검색 결과 노출 여부는 해당 사용자의 **권한 보유 여부**로만 판단합니다.
#### 6-19-6-1. Override 문서 버전 복원 시 권한 처리

버전 히스토리(오픈 이후 예정)를 통해 이전 버전으로 복원할 경우:

| 상황 | 처리 방식 |
|---|---|
| Override 된 문서를 이전 버전으로 복원 | 문서 내용만 복원 · **Override Flag 유지** (권한은 복원 대상 아님) |
| Override 없던 시점 버전으로 복원 | 내용 복원 + Override Flag는 **현재 상태 유지** (버전 복원이 권한 초기화하지 않음) |
| 복원 주체 | Override 해제 권한자(소유자/최고관리자)만 복원 가능 |

> **원칙:** 버전 복원은 문서 콘텐츠에 대한 작업. 권한(Override Flag)은 별도 관리되며 복원에 영향받지 않습니다.

#### 6-19-7. Override = 개별 권한 보호 원칙

Override가 설정된 문서는 **폴더 편집자도 제어할 수 없습니다.** 이는 민감 문서의 보안을 강화하는 핵심 원칙입니다.

**폴더 편집자의 한계:**

| 행위 | Override 없는 문서 | Override된 문서 |
|---|---|---|
| 내용 편집 | 가능 | **불가** |
| 이동 | 가능 | **불가** |
| 삭제 | 가능 | **불가** |
| 이름 변경 | 가능 | **불가** |
| 폴더 내부에서 보이는가 | 정상 노출 | 🔒 표시 + 제어 차단 |

**폴더 편집자가 Override 문서에 접근 시도할 때:**

> "이 문서는 별도 권한이 설정되어 있어요. 소유자에게 권한 요청이 필요합니다."

**Override 해제 권한자:**

| 역할 | 해제 가능 |
|---|---|
| 문서 소유자 | O |
| 최고 관리자 | O (감사 로그 기록) |
| 문서 편집 권한 있는 사람 | O (단, 공유된 Override면 제한 있을 수 있음) |
| 폴더 편집자 | **X** (Override 문서 제어 불가 · 해제도 불가 — 해제는 소유자/최고관리자만) |

**설계 의도:**

- Override = "이 문서는 특별히 관리된다" 는 명시적 표현
- 개별 권한 보호 원칙이 없으면 Override의 의미가 사라짐 (폴더 편집자가 덮어쓸 수 있으면)
- 회사의 민감 정보 · 개인정보 · 전략 문서 등 보호에 필수

#### 6-19-8. 회사 도서관 예외 정책

회사 도서관은 **전 직원 공개 원칙**을 유지하기 위한 예외가 있습니다.

| 권한 유형 | 회사 도서관 규칙 |
|---|---|
| **조회** | 전 직원 고정 · **Override로 좁힐 수 없음** |
| **편집** | 폴더/문서 레벨에서 Override 가능 (일반 원칙 적용) |

**조회 Override가 금지되는 이유:**

- 회사 도서관은 "공식 문서·기준 문서·공지" 중심 공간
- 조회 제한을 허용하면 전사 공유 원칙이 무너짐
- 기밀 문서는 **회사 도서관이 아닌 공유 서재에 위치**해야 함

**대체 플로우:**

| 요구 사항 | 해결 방안 |
|---|---|
| "일부에게만 공개하고 싶다" | 회사 도서관이 아닌 공유 서재 사용 |
| "조회는 전사, 편집은 특정 팀만" | 회사 도서관 + 편집 Override 가능 |
| "임원만 보는 문서" | 공유 서재 "임원진" 등 별도 서재 생성 |

**UI에서의 표현:**

회사 도서관 문서의 공유 팝업에서:
- 조회 범위: "전 직원 (고정)" 안내 · 변경 불가
- 편집 권한: 개별 설정 가능

#### 6-19-9. 레이어별 적용 정리

| 레이어 | 폴더 조회 | 폴더 편집 | 문서 조회 Override | 문서 편집 Override |
|---|---|---|---|---|

| 🏢 **회사 도서관** | 전 직원 (고정) | 설정 가능 | **불가 (예외)** | 가능 |

## 8. 폴더 간 문서 이동 권한 정책

> v1.2 신규 추가

### 8-1. 정책 개요

**목적:** 문서를 다른 폴더로 이동할 때 발생하는 권한 변경을 사용자에게 명확히 안내하고, 위험도에 따라 경고 수준을 분기하여 의도치 않은 권한 변경 사고를 방지합니다.

**배경 (현행 문제점):**
- 현재 폴더 간 문서 이동 시 "폴더의 권한을 따르시겠습니까?" 단일 문구만 표시되어, 권한이 넓어지는지 좁아지는지 사용자가 판단할 수 없음
- 모든 이동 상황에 동일한 UI가 적용되어, "공유 서재 → 회사 도서관" 확대와 "회사 도서관 → 공유 서재" 축소가 동일한 위험도로 표시됨
- 외부 공개 폴더로의 이동에도 별도 경고가 없어 사내 문서가 실수로 외부 공개될 위험이 있음

### 8-2. 위험도 분류 기준

문서 이동 시 권한 변경의 위험도를 3단계로 분류합니다.

| 위험도 | 조건 | 색상 | 버튼 스타일 |
|---|---|---|---|
| 안전 (Safe) | 접근 범위가 좁아지는 경우 | 초록 | 기본 보라색 |
| 주의 (Warning) | 접근 범위가 넓어지거나 권한 수준이 올라가는 경우 | 노란색 | 주황색 |
| 위험 (Danger) | 외부 공개로 변경되는 경우 | 빨간색 | 빨간색 |

**위험도 판정 로직:**

| 현재 → 이동 | 판정 |
|---|---|
| 나만 → 공유 서재 | 주의 (범위 확대) |
| 나만 → 회사 도서관 | 주의 (범위 확대) |
| 공유 서재 → 회사 도서관 | 주의 (범위 확대) |
| 회사 도서관 → 공유 서재 | 안전 (범위 축소) |
| 회사 도서관 → 나만 | 안전 (범위 축소) |
| 공유 서재 → 나만 | 안전 (범위 축소) |
| 공유 서재 A → 공유 서재 B | 주의 (서재 멤버 구성 변경) |
| 내부(나만/공유 서재/회사 도서관) → 링크가 있는 누구나 | 위험 (외부 공개) |
| 링크가 있는 누구나 → 내부 | 안전 (외부 차단) |
| 동일 범위, 조회 → 편집 | 주의 (권한 상승) |
| 동일 범위, 편집 → 조회 | 안전 (권한 축소) |

### 8-3. 이동 확인 팝업 UI 구성

모든 폴더 간 문서 이동 시 아래 구성의 확인 팝업을 표시합니다.

**팝업 구성 요소:**

| 구성 | 설명 |
|---|---|
| 제목 | "문서 이동" |
| 서브 텍스트 | "{문서명}을 다른 폴더로 이동합니다." |
| before → after 카드 | 현재 폴더명+권한과 이동할 폴더명+권한을 나란히 표시 |
| 위험도 안내 바 | 위험도별 색상 + 메시지 (아래 참조) |
| 변경 항목 리스트 | 접근 범위 변경, 권한 수준 변경을 한 줄씩 before → after로 표시 |
| 액션 버튼 | [기존 권한 유지] / [폴더 권한 적용] (위험도별 색상 분기) |

**위험도별 안내 메시지:**

| 위험도 | 메시지 |
|---|---|
| 안전 | "접근 범위가 좁아집니다. 이동 후 {대상 조직명} 멤버만 이 문서를 볼 수 있어요." |
| 주의 | "접근 범위가 넓어집니다. 이동 후 {대상} 직원이 이 문서를 {권한}할 수 있어요." |
| 위험 | "외부 공개됩니다. 이동 후 링크가 있는 누구나(외부인 포함) 이 문서에 접근할 수 있어요. 민감한 내용이 포함되어 있지 않은지 확인해주세요." |

### 8-4. 변경 항목 표시 규칙

| 변경 유형 | 아이콘 | 색상 | 표시 형식 |
|---|---|---|---|
| 범위 축소 | - | 초록 | 접근 범위: {이전 범위(N명)} → {이후 범위(N명)} |
| 범위 확대 | + | 노란색 | 접근 범위: {이전 범위(N명)} → {이후 범위(N명)} |
| 외부 공개 | ! | 빨간색 | 접근 범위: {이전 범위(N명)} → 외부 공개 (누구나) |
| 권한 유지 | = | 파란색 | 권한 수준: {권한} 유지 |
| 권한 상승 | + | 노란색 | 권한 수준: {이전} → {이후} |
| 권한 축소 | - | 초록 | 권한 수준: {이전} → {이후} |

인원수는 해당 조직/범위에 속한 멤버 수를 실시간으로 조회하여 표시합니다.

### 8-5. 권한 상속 규칙

| 상황 | 처리 |
|---|---|
| "폴더 권한 적용" 선택 | 문서 권한이 이동할 폴더의 권한으로 변경됨 |
| "기존 권한 유지" 선택 | 문서의 기존 권한이 그대로 유지됨 (폴더 권한과 독립) |

### 8-6. 하위 문서 포함 이동 시 추가 확인

폴더 자체를 이동하거나 하위 문서가 포함된 경우, 팝업에 추가 안내를 표시합니다.

| 항목 | 내용 |
|---|---|
| 추가 안내 | "이 폴더에는 {N}개의 하위 문서가 포함되어 있습니다. 모든 하위 문서에도 동일한 권한 변경이 적용됩니다." |
| 하위 문서 미리보기 | 영향받는 하위 문서 목록 최대 5개 표시 + "외 N개" |

### 8-7. 이동 대상 선택 UI (공유 서재 2단계 선택)

> v2.9 신규 추가

문서/폴더를 "공유 서재로 이동" 선택 시, 구체적으로 **어떤 공유 서재**로 이동할지 2단계로 선택합니다.

**이동 대상 선택 플로우:**

| 회사 도서관 | 전 직원 허용 · 최고 관리자만 최상단 폴더 이동 가능 |

**폴더 선택 옵션:**

| 옵션 | 동작 |
|---|---|
| 서재 루트에 이동 | 하위 폴더 없이 서재 최상단으로 |
| 특정 폴더 선택 | 서재 내 폴더 트리에서 선택 (2 depth까지 펼침) |
| 새 폴더 만들기 | 선택 화면에서 [+ 새 폴더] 버튼 · 즉시 생성 후 이동 |

**확정 후 동작:**
- 대상 선택 완료 → 위험도 판정(8-2) 수행
- 위험도 결과에 따라 기존 확인 팝업(8-3) 진입
- 최종 확인 후 이동 실행

## 9. 내보내기(PDF/DOCX) 품질 기준

> v1.3 신규 추가

### 9-1. 정책 개요

**목적:** 위키 문서를 PDF/DOCX로 내보낼 때 객체별 변환 규칙과 페이지 레이아웃 기준을 정의하여, 내보내기 결과물의 품질을 일관되게 보장하고 QA 검수 기준을 제공합니다.

**배경:** 블록 에디터의 콘텐츠를 워드/PDF로 변환할 때 표, 코드블록, 이미지 레이아웃, 체크박스 등에서 서식 깨짐이 발생합니다. 변환 품질 기준이 없으면 QA도 "이 정도면 괜찮은지" 판단할 수 없고, 사용자에게 "내보내기했더니 표가 깨진다"는 VOC가 반복됩니다.

### 9-2. 페이지 레이아웃

| 항목 | PDF | DOCX |
|---|---|---|
| 용지 크기 | A4 (210x297mm) | A4 기본, 다운로드 후 변경 가능 |
| 여백 | 상하좌우 20mm | 상하좌우 25.4mm (워드 기본값) |
| 헤더 | 문서 제목 (좌측) + 페이지 번호 (우측) | 동일 |
| 푸터 | "Flow Wiki에서 내보냄 · {YYYY.MM.DD}" | 동일 |
| 페이지 번호 | 우측 상단 "1/N" 형식 | 동일 |
| 기본 폰트 | Pretendard (한글) + Inter (영문), 폴백: Noto Sans KR | 맑은 고딕 (한글) + Arial (영문) |
| 본문 크기 | 11pt | 11pt |
| 제목 크기 | H1: 22pt / H2: 18pt / H3: 15pt | 동일 (Word Heading 1~3 스타일 매핑) |
| 줄간격 | 1.6 배수 | 1.5 배수 (워드 관행에 맞춤) |

### 9-3. 파일명 및 제한

| 항목 | 규칙 |
|---|---|
| 파일명 형식 | `{문서제목}_{YYYYMMDD}.pdf` / `{문서제목}_{YYYYMMDD}.docx` |
| 문서 제목 치환 | 특수문자(`/ \ : * ? " < > \|`)는 `_`로 치환 |
| 파일명 최대 길이 | 100자 (초과 시 뒤에서 말줄임) |
| 최대 페이지 수 | 200페이지 초과 시 경고 토스트 + "문서를 분할하여 내보내시겠습니까?" 제안 |
| 최대 파일 크기 | 50MB 초과 시 이미지 압축 적용 후 재시도 안내 |

### 9-4. 객체별 변환 규칙

| 객체 | PDF 변환 기준 | DOCX 변환 기준 | QA 위험도 |
|---|---|---|---|
| 텍스트 (B/I/U/S) | 서식 100% 유지 | 서식 100% 유지 | 낮음 |
| 제목 (H1/H2/H3) | 폰트 크기·굵기 유지 | Word Heading 1~3 스타일 매핑 | 낮음 |
| 순서/불릿 목록 | 들여쓰기 + 번호/불릿 유지, 최대 3뎁스 | 네이티브 Word 목록으로 변환 | 중간 |
| 체크박스 | ☑/☐ 문자 변환 + 취소선(완료 시) | 네이티브 체크박스 컨트롤 | 중간 |
| 표 (Table) | 셀 병합·배경색·테두리 유지, 페이지 넘김 시 헤더 행 반복 | 네이티브 Word 표로 변환, 셀 병합 유지 | 높음 |
| 코드블록 | 배경색(#F5F5F5) + 고정폭 폰트(Consolas/Menlo), 줄바꿈 허용 | 음영 박스 + 고정폭 폰트, 구문 강조 색상 유지 | 중간 |
| 인용 | 좌측 보더(4px, #CCCCCC) + 들여쓰기 16px | Word 인용 스타일 매핑 | 낮음 |
| 토글 | 모두 펼친 상태로 출력 (접기 기능 없음), 토글 제목은 볼드 처리 | 동일 | 중간 |
| 이미지 | 원본 해상도 유지, 페이지 폭 초과 시 자동 리사이즈, 캡션 유지 | 인라인 이미지 삽입, 150dpi 이상 보장 | 높음 |
| 파일 첨부 | 파일명 + 용량 텍스트 표시 (첨부파일 자체는 미포함) | 동일 | 낮음 |
| 플로우 업무 카드 | 텍스트 요약 변환: 업무명 / 상태 / 담당자 / 마감일 | 동일 구조로 텍스트 박스 변환 | 중간 |
| 플로우 게시글 카드 | 게시글 제목 + 작성자 + 날짜 텍스트 변환 | 동일 | 중간 |
| 텍스트 색상 | 색상 유지 (RGB 값 그대로) | 색상 유지 | 중간 |
| 형광펜 색상 | 배경색 유지 | Word 형광펜 매핑 (가장 가까운 색상) | 중간 |
| 페이지 내부 링크 | 텍스트만 표시 + 원본 URL 각주 삽입 | 동일 | 낮음 |
| 외부 링크 | 클릭 가능한 하이퍼링크 (파란색 밑줄) | 동일 | 낮음 |
| 커버 이미지 | 첫 페이지 상단에 전체 폭으로 삽입 | 동일 | 중간 |
| 아이콘/이모지 | PNG로 래스터화하여 인라인 삽입 (16x16px) | 동일 | 중간 |

### 9-5. 표(Table) 변환 상세 규칙

표는 내보내기 시 가장 깨지기 쉬운 객체이므로 별도 규칙을 정의합니다.

| 항목 | 규칙 |

| 최대 열 수 | A4 기준 최대 8열까지 정상 렌더링 보장. 9열 이상은 폰트 9pt로 축소 후 출력, 그래도 초과 시 가로 방향(Landscape) 전환 안내 |

| 10 | 페이지 레이아웃 | 여백, 헤더, 푸터, 페이지 번호가 규칙대로 표시되는가 |
| 11 | 파일명 | 특수문자 치환, 날짜 형식이 규칙대로 생성되었는가 |
| 12 | 한글 깨짐 | 폰트 임베딩이 정상이고 한글/특수문자가 깨지지 않는가 |

## 10. 에디터 UI 정책

### 10-1. 문서 제목

| 항목 | 정책 |
|---|---|
| Placeholder | 제목 비었을 때 "새 페이지" 표시. 저장되지 않음. |
| 줄바꿈 | 제목 내 줄바꿈 허용. 최대 글자 수 제한 없음. |
| 사이드바 노출 | 1줄 말줄임 처리. |
| Enter / ↓ | 본문 첫 블록으로 포커스 이동. |
| ↑ (본문에서) | 제목으로 포커스 이동. |
| 실시간 연동 | 입력 즉시 (debounce 300ms) 사이드바 문서명 / 브라우저 탭 타이틀 / 검색 결과 반영. |

**제목 없을 때 저장 차단:**

제목이 비어있는 상태에서 저장을 시도하면 저장을 차단합니다.

| 항목 | 정책 |
|---|---|
| 자동 저장 | 제목 입력 전까지 비활성 |
| 저장 시도 시 | 제목 영역 빨간 테두리 + "제목을 입력해주세요" 안내 문구 노출 |
| 저장 가능 조건 | 제목 1자 이상 입력 시 저장 활성화 |

### 10-2. 커버 이미지

| 항목 | 정책 |
|---|---|
| 위치 | 에디터 최상단 full-width. 고정 높이 200px. |
| 추가 버튼 | 커버 없을 때 "🖼 커버 추가" 노출. 커버 있으면 미노출. |
| hover 시 | "변경 / 제거" 버튼 우하단 노출. |
| 갤러리 | 기본 제공 12종 (그라디언트/단색/이미지 포함). 향후 확장 가능. |
| 이미지 업로드 | JPG / PNG / WebP, 최대 5MB. 드래그앤드롭 지원. |
| 저장 시점 | "적용" 클릭 시 자동 저장. 제거는 즉시 저장. |

### 10-3. 문서 아이콘

| 항목 | 정책 |
|---|---|
| 추가 버튼 | 아이콘 없을 때 "🙂 아이콘 추가" 노출. 아이콘 있으면 미노출. |
| 크기 (커버 없음) | 제목 바로 위 64px. |
| 크기 (커버 있음) | 커버 하단 -36px 겹침. 흰 배경 + 그림자 처리. |
| 이모지 피커 | 카테고리 8종. 한글/영문 검색 지원. 선택 즉시 반영 + 자동 저장. |
| 연동 | 사이드바 문서명 앞 16px / 에디터 제목 위 64px / 브라우저 탭 파비콘 대체. |
| 저장 시점 | 선택/제거 즉시 자동 저장. |

### 10-4. 자동 저장

| 상태 | 조건 | 표시 |
|---|---|---|
| 저장 중 | API 요청 중 | 보라 점 + "저장 중…" |
| 저장됨 | 성공 응답 | 초록 점 + "저장됨" |
| 저장 실패 (1~2회) | 네트워크 오류 / 5xx | 빨간 점만 노출. 배너 미노출. |
| 저장 실패 (3회 이상) | 연속 3회 이상 실패 | 빨간 점 + 상단 배너 노출. 문구: "3회 이상 저장에 실패했어요. 문서를 복제하여 저장하시는 것을 추천합니다." 버튼: 문서 복제. |
| 오프라인 | navigator.onLine = false | 회색 점 + 상단 배너 |

- 저장 트리거: 입력 후 **2초 debounce** 자동 저장. 포커스 이탈(blur) 시 즉시 저장.
- 저장 성공 시 실패 카운트 초기화. 배너 자동 해제.
- 오프라인 중 변경 내용 로컬 임시 보관. 연결 복구 즉시 자동 저장 재시도.
- 오프라인 중 이동/공유 버튼 비활성.

### 10-5. 불러오기

| 항목 | 정책 |
|---|---|
| 버튼 위치 | 에디터 하단 중앙 고정 (position: fixed). |
| UI | 검색 팝업과 동일한 컴포넌트 재사용. |
| 검색 범위 | 내가 조회 권한 있는 모든 문서. |
| 삽입 동작 | 현재 커서 위치에 선택 문서 전체 내용 인라인 삽입. 삽입 후 팝업 자동 닫힘. |
| 권한 없는 문서 | 결과에 노출되나 선택 불가 (dim + 🔒 + "권한 없음" 배지). |
| 자기 자신 | 검색 결과에서 제외. |
| 삭제/휴지통 문서 | 검색 결과에서 제외. |

### 10-6. 더보기 메뉴

| 순서 | 항목 | 동작 |
|---|---|---|
| 1 | 문서 복제하기 | 같은 폴더에 "{원제목} 사본" 생성 후 이동 |
| 2 | 링크 복사 | URL 복사 + "링크가 복사되었습니다" 토스트. **노출 조건: 어드민 외부 공유 링크 관리에서 전체 문서 외부 공유 ON 시에만 메뉴에 노출** (16-7 참조) |
| 3 | PDF 내보내기 | 변환 후 다운로드. 파일명: {문서제목}.pdf |
| 4 | 인쇄 | 브라우저 인쇄 다이얼로그 호출. 사이드바/버튼 숨김 CSS 적용 |
| 5 | 삭제 | 확인 모달 후 휴지통 이동. 동시 편집자 있으면 차단 (6-15 정책) |

### 10-7. 링크 복사 토스트 정책

링크 복사 동작이 발생하는 모든 진입점에서 완료 토스트를 노출한다.

- 토스트 자동 소멸: 2초 후
- 버튼명 통일: **링크복사 → 링크 복사** (띄어쓰기 통일)

### 10-8. 본문 이미지 블록

|---|---|

| 내보내기 시 | PDF/DOCX는 페이지 폭(170mm) 초과 시 자동 리사이즈 (에디터 표시와 별개). |

## 12. 붙여넣기 소스별 정상 동작 정리

클립보드 `text/html` 직렬화 이슈로 인한 소스별 태그 노출 버그 기준 정의.

| 붙여넣기 대상 | 포맷 수용 | 정상 동작 | 태그 노출 시 판단 |
|---|---|---|---|
| 메모장 (Windows Notepad) | text/plain only | 순수 텍스트만 표시, 서식 제거 | 정상 (태그 노출 X) |
| VSCode / Sublime 등 코드 에디터 | text/plain only | 순수 텍스트만 표시 | 정상 (태그 노출 X) |
| MS Word | text/html 우선 | 서식 유지되어 표시 | **버그** |
| Google Docs | text/html 우선 | 서식 유지되어 표시 | **버그** |
| Gmail / Outlook 메일 작성창 | text/html 우선 | 서식 유지되어 표시 | **버그** |
| Slack 메시지 입력창 | text/html 부분 수용 | 볼드·링크 등 기본 서식 유지 | **버그** |
| 노션 / 컨플루언스 / 타 위키 | 자체 포맷 변환 | 블록 구조로 재구성 | **버그** |
| 카카오톡 / 텔레그램 채팅창 | text/plain 위주 | 순수 텍스트로 표시 | 정상 (태그 노출 X) |
| 브라우저 주소창 / 검색창 | text/plain only | 순수 텍스트만 표시 | 정상 (태그 노출 X) |

**핵심 판단 기준:** 리치 텍스트를 수용하는 에디터(Word, Docs, 메일, 협업툴)에서 HTML 태그 문자열이 그대로 보이면 프론트의 클립보드 `text/html` 직렬화 이슈로 확정. plain text만 받는 앱(메모장, 코드 에디터, 채팅앱)은 태그가 안 보이는 게 정상이므로 버그 재현 대상에서 제외.

---

1. **렌더링 안정성:** 허용되지 않는 중첩 구조를 원천 차단하여 HTML 태그 노출 및 레이아웃 깨짐 현상을 방지합니다.
2. **편집 일관성:** / 메뉴와 붙여넣기 두 진입점에 동일한 규칙을 적용하여 사용자가 예측 가능한 편집 환경을 경험할 수 있습니다.
3. **데이터 보호:** 차단 객체를 plain text로 변환함으로써 내용 손실 없이 안전하게 처리합니다.
4. **QA 이슈 해소:** 제목 블록 내 번호 입력 불가 등 최다 빈출 이슈를 블록 타입별 자동 변환 분기로 근본적으로 해결합니다.
5. **탐색 효율화:** 문서 공개 범위 3단계 + 사이드바 스코프 필터로 관련 없는 문서 노출을 차단하여 탐색 비용을 대폭 줄입니다.
6. **공유 서재 단위 문서 관리:** 전 직원이 자유롭게 공유 서재를 생성하고 초대로 참여하여, 조직도와 무관하게 목적에 맞는 문서 관리 환경을 제공합니다.
7. **권한 변경 사고 방지:** 폴더 간 문서 이동 시 위험도별 분기 경고와 before→after 미리보기로 의도치 않은 외부 공개, 권한 확대 사고를 사전에 차단합니다.
8. **내보내기 품질 보장:** 객체별 변환 규칙과 QA 체크리스트를 정의하여 PDF/DOCX 내보내기 결과물의 품질을 일관되게 보장하고, "표가 깨진다" 류의 VOC를 근본적으로 해소합니다.

## 13. 즐겨찾기 페이지 권한 정책

### 13-1. 개요

사용자가 즐겨찾기한 페이지의 조회 권한이 사라진 경우, 즐겨찾기가 노출되는 모든 위치에서 일관되게 숨김 처리한다.

### 13-2. 적용 대상 화면

| 노출 위치 | 처리 방식 |
|---|---|
| 위키 홈 > 즐겨찾기 모아보기 | 숨김 |
| 사이드메뉴 > 즐겨찾기 탭 | 숨김 |
| 기타 즐겨찾기 노출 영역 (향후 추가 포함) | 숨김 |

### 13-3. 권한 상실 발생 케이스

다음 상황에서 권한이 사라진 것으로 간주한다.

- 페이지 소유자가 해당 사용자의 조회 권한을 회수한 경우
- 즐겨찾기된 페이지가 사용자가 접근 권한 없는 폴더/서재로 이동된 경우
- 페이지가 전체 공개 → 제한 공개로 전환되며 해당 사용자가 제외된 경우
- 사용자가 속한 그룹이 해당 페이지의 권한 그룹에서 제외된 경우

### 13-4. 처리 동작

| 항목 | 동작 |
|---|---|
| 즐겨찾기 데이터 | 유지 (DB에서 삭제하지 않음) |
| 화면 노출 | 숨김 (목록에서 제외) |
| 권한 재부여 시 | 즐겨찾기 목록에 자동 재노출 |
| 페이지 자체가 삭제된 경우 | 즐겨찾기에서 제거 |

### 13-5. 예외 처리

- **권한 확인 API 호출 실패 시:** 기존 즐겨찾기 목록을 그대로 유지하여 사용자 경험 저하 방지
- **페이지가 완전 삭제된 경우:** 즐겨찾기 데이터도 함께 제거

### 13-6. 담당 범위

| 영역 | 담당 |
|---|---|
| 즐겨찾기 조회 API 권한 필터링 | 백엔드 |
| 모든 즐겨찾기 노출 화면의 일관 처리 | 프론트엔드 |

## 14. 플로우 업무 양방향 연동 정책

> v2.0 신규 추가

### 14-1. 정책 개요

**목적:** 위키 내 플로우 업무 블록을 단순 임베드(iframe 읽기 전용)에서 **양방향 편집 객체**로 업그레이드하여, 문서 맥락 안에서 업무를 바로 수정·관리할 수 있게 합니다.

**배경 (현행 문제점):**
- 플로우 업무가 iframe으로만 노출되어 수정이 불가능하고, 플로우에서의 변경사항은 새로고침해야 반영됨
- 회의록·프로젝트 문서에서 업무 상태를 확인하고 수정하려면 플로우로 이동해야 하는 컨텍스트 스위칭 발생
- 위키와 플로우 간 데이터 단절로 업무 상태 불일치 빈번

**핵심 원칙:** 위키에서의 편집은 즉시 플로우에 반영되고, 플로우에서의 변경은 실시간으로 위키에 반영됩니다. 두 서비스는 단일 데이터 소스를 공유하며 UI만 다르게 표현합니다.

### 14-2. 블록 생성 플로우

슬래시 메뉴 `/플로우 업무` 또는 `@업무` 트리거로 생성합니다.

| 단계 | 동작 |
|---|---|
| 1. 진입 | `/` → "플로우 업무" 선택 또는 `@업무` 자동완성 |
| 2. 검색 | 프로젝트 · 업무명 · 담당자명 검색 (최근 조회 프로젝트 상단 고정) |
| 3. 선택 | 단일 업무 선택 · 복수 선택 모두 지원 (복수 시 각각 별도 블록 생성) |
| 4. 삽입 | 에디터 본문에 업무 블록 삽입. 기본값은 `카드 뷰` |
| 5. 뷰 전환 | 블록 우측 ···  메뉴에서 `카드 뷰` / `상세 뷰` / `체크박스 뷰` 전환 가능 |

### 14-3. 편집 가능 필드

위키에서 편집 가능한 필드는 플로우 업무의 전체 필드와 일치합니다.

| 필드 | 타입 | 위키 편집 | 양방향 동기화 |
|---|---|---|---|
| 제목 | 텍스트 | O | O |
| 상태 | select (할 일/진행중/완료 등) | O | O |
| 담당자 | 멤버 선택 | O | O |
| 마감일 | 날짜 | O | O |
| 시작일 | 날짜 | O | O |
| 우선순위 | select (긴급/높음/보통/낮음) | O | O |
| 프로젝트 | 읽기 전용 | X | — |
| 하위 업무 | 리스트 | O (추가/제거) | O |
| 댓글 | 텍스트 | O (작성만) | O |
| 첨부파일 | 파일 | O (추가/제거) | O |
| 태그 | 멀티 select | O | O |
| 진척도 | % | O | O |
| 업무 ID | 읽기 전용 | X | — |

**프로젝트 필드는 플로우에서만 변경 가능합니다.** 위키에서 업무를 다른 프로젝트로 이동하려면 "플로우에서 열기"를 통해 이동하도록 안내합니다.

### 14-4. 권한 모델

위키에서 플로우 업무를 수정하려면 **두 권한 모두** 필요합니다 (AND 조건).

| 권한 | 필요 이유 |
|---|---|
| 위키 문서 편집 권한 | 문서에 삽입된 블록을 변경하는 행위 자체의 권한 |
| 플로우 업무 수정 권한 | 원본 데이터를 변경하는 행위의 권한 |

**권한별 UI 상태:**

| 상황 | 표시 |
|---|---|
| 두 권한 모두 있음 | 정상 편집 UI (필드 클릭 시 편집 모드) |
| 위키 문서 편집만 있음 | 업무 블록 "읽기 전용" 표기 + 편집 시도 시 "플로우 업무 권한이 필요합니다" 안내 |
| 플로우 업무 권한만 있음 | 문서 자체가 조회 전용이므로 블록도 조회 전용 |
| 둘 다 없음 | 읽기 전용 (기본 상태) |

**플로우 업무 수정 권한 기준 (플로우 정책 따름):**
- 업무 생성자
- 업무 담당자
- 업무 참여자 (watcher)
- 프로젝트 관리자
- 최고 관리자

### 14-5. 양방향 동기화 정책

**동기화 방식:** WebSocket 기반 실시간 동기화를 기본으로 하고, 연결 끊김 시 30초 폴링으로 자동 폴백하는 **하이브리드 방식**입니다.

**모드별 동작:**

| 모드 | 조건 | 업데이트 주기 |
|---|---|---|
| WebSocket (기본) | 연결 정상 | 이벤트 즉시 (push) |
| 폴링 폴백 | WebSocket 재연결 5회 실패 | 30초 간격 |
| 재연결 시도 | 폴링 모드에서도 5분마다 WebSocket 재연결 시도 | 성공 시 WebSocket 모드 복귀 |
| 오프라인 | `navigator.onLine = false` | 편집 불가 · 배너 표시 |

**동기화 트리거 조건:**
- **문서가 열려 있을 때만** 동기화 활성화. 탭이 백그라운드인 경우 동기화 중지 (리소스 절약).
- 탭 포커스 복귀 시 즉시 최신 상태 가져오기 (full fetch).

**이벤트별 지연 목표:**

| 이벤트 | 동기화 방향 | WebSocket 지연 | 폴링 지연 |
|---|---|---|---|
| 위키에서 필드 변경 | 위키 → 플로우 | 500ms 이내 | 저장은 즉시, 반영 확인은 최대 30초 |
| 플로우에서 필드 변경 | 플로우 → 위키 | 1초 이내 | 최대 30초 |
| 위키 블록 추가 | 위키 → 플로우 (watcher 추가) | 즉시 | 즉시 (저장 자체는 독립) |
| 위키 블록 삭제 | 블록만 제거 · 플로우 업무는 유지 | 즉시 | 즉시 |
| 플로우 업무 삭제 | 플로우 → 위키 ("삭제된 업무" 표시) | 1초 이내 | 최대 30초 |

**모드 전환 안내:**
- 폴링 모드 진입 시 상단 배너 "실시간 연결이 끊어져 30초 주기로 업데이트됩니다" 노출
- WebSocket 복귀 시 배너 자동 해제 + 토스트 "실시간 연결이 복구되었습니다"

**블록 삭제 ↔ 업무 삭제 구분:**
- 위키 블록 삭제: 문서에서 참조만 끊음. 플로우 업무는 그대로 유지.
- 플로우 업무 삭제: 위키 블록은 "삭제된 업무" 상태로 전환 (회색 + 취소선).

### 14-6. 충돌 처리 (Conflict Resolution)

**충돌 감지 조건:** 동일 필드가 위키와 플로우에서 5초 이내 동시에 다른 값으로 변경될 때.

**감지 원리:** 각 필드에 `updated_at` 타임스탬프 + `version` 증분값을 유지. 저장 시 version 불일치 감지.

**처리 방식 (수동 해결 원칙):**

| 상황 | UI 동작 |
|---|---|
| 충돌 감지 | 해당 필드 주황 테두리 + ⚠️ 아이콘 |
| 클릭 시 모달 | "동시 수정이 감지되었습니다" 모달 노출 |
| 모달 구성 | 내 변경값 / 다른 사람 변경값 / 현재 서버값 3개 비교 |
| 선택 옵션 | [내 값 적용] / [상대 값 적용] / [내가 다시 작성] |
| 해결 전 편집 | 해당 필드만 잠금 (다른 필드는 정상 편집 가능) |

**알림:** 충돌 감지 시 토스트 "다른 사용자와 동시 편집 중입니다. 해결이 필요합니다." + 해당 블록으로 자동 스크롤.

### 14-7. 리스트 뷰

플로우 업무 블록은 **리스트 뷰 단일 형식**으로 표현됩니다. 상세 정보가 필요한 경우 사이드 패널(14-12)로 오픈합니다.

**컬럼 구성 (전체 고정값 · overflow 가로 스크롤):**

| 컬럼 | 고정 너비 | 내용 | 편집 가능 | 비고 |
|---|---|---|---|---|
| 프로젝트명 | **160px** | 프로젝트별 색상 배지 · 읽기 전용 | X | 좌측 sticky 고정 |
| 업무명 | **320px** | 업무 제목 · overflow ellipsis + **fade-out 말줄임** | O (더블클릭 인라인) | 넘칠 때 끝에 투명도 그라디언트 처리 |
| 상태 | **110px** | 배지 (요청·개발중·완료 등) | O (클릭 드롭다운) | 중앙 정렬 |
| 담당자 | **120px** | 이름 · 복수 시 +N | O (클릭 팝오버) | — |
| 시작일 | **90px** | MM/DD · 경과 시 빨간색 | O (클릭 date picker) | 중앙 정렬 |
| 마감일 | **90px** | MM/DD · 미설정 시 — | O (클릭 date picker) | 중앙 정렬 |
| 우선순위 | **100px** | 배지 (긴급·높음·보통) | O (클릭 드롭다운) | 중앙 정렬 |
| 진척도 | **110px** | 프로그레스 바 + % 숫자 | O (클릭 슬라이더) | — |
| 작성자 | **100px** | 이름만 표시 · 읽기 전용 | X | — |
| **합계** | **1200px** | | | overflow-x: auto · 기준 1440px (본문 1172px) |

**프로젝트명 컬럼 정책:** 위키 문서 하나에 여러 프로젝트 업무가 혼재할 수 있어 좌측 sticky 고정. 플로우에서만 변경 가능하므로 위키에서는 읽기 전용 배지로만 표시.

**스크롤 및 높이 정책:**

| 항목 | 정책 | 근거 |
|---|---|---|
| 행 높이 | 44px (padding 11px + 내용 22px + 여백) | 컴팩트 + 터치 타겟 44px 기준 |
| 헤더 높이 | 38px | sticky 고정 |
| **최소 높이** | **126px** (헤더 + 2행) | 업무 1~2개여도 빈 블록 방지 |
| **최대 높이** | **480px** (헤더 + 10행) | 에디터 가용 높이의 약 2/3, 블록 위아래 콘텐츠 여유 확보 |
| 자동 확장 | 126~480px 구간에서 업무 수에 따라 자동 | — |
| 가로 스크롤 | 전체 컬럼 합계 **1200px** · overflow-x: auto · 프로젝트명 좌측 sticky | 기준 1440px · 본문 1172px · 1200px 초과 시 자연스러운 가로 스크롤 |
| 프로젝트명 컬럼 | **좌측 고정 (sticky left:0, 160px)** | 가로 스크롤 중 프로젝트 식별 유지 |
| 헤더 행 | **상단 고정 (sticky top:0)** | 세로 스크롤 중 컬럼 식별 유지 |
| 초과 시 | 10개 초과 → 내부 세로 스크롤 발생 | 페이지 전체 스크롤과 분리 |

**프로젝트명 컬럼 정책:** 업무명과 분리된 **독립 컬럼(160px)**으로 표시. 배지 형태로 노출하며 가로 스크롤 시 좌측 sticky 고정. 플로우에서만 변경 가능하므로 위키에서는 **읽기 전용**. 같은 프로젝트의 업무는 컬럼 정렬 기본값 기준으로 자동 그룹화.

**정렬 모드:**

| 모드 | 동작 |
|---|---|
| **컬럼 정렬 (기본)** | 프로젝트명 1차 정렬 + 선택한 컬럼 2차 정렬. 헤더 클릭으로 ▲/▼ 전환 |
| **직접 정렬** | 사용자가 D&D로 행 순서 수동 변경. 활성화 시 컬럼 정렬 자동 해제. 순서 저장 후 플로우 동기화 |
| 모드 토글 | 블록 상단 우측 "컬럼 정렬 / 직접 정렬" 세그먼트 버튼 |
| 정렬 상태 유지 | 문서별 · 블록별 독립 저장 |

**상세 정보 확인:** 업무 행 클릭 → **중앙 레이어 팝업**으로 전체 필드 · 하위 업무 · 댓글 확인 및 편집 (섹션 14-12 참고).

### 14-8. 삭제된 업무 처리

플로우에서 원본 업무가 삭제(휴지통 이동 · 영구 삭제)된 경우의 처리입니다.

| 상황 | 블록 표시 |
|---|---|
| 휴지통으로 이동됨 | "삭제됨" 배지 + 회색 처리 + [블록 제거] 버튼 |
| 영구 삭제됨 | "삭제된 업무" 플레이스홀더 + [블록 제거] 버튼 |
| 접근 권한 상실 | "접근 권한 없음" 표시 (버튼 없음, 위키 블록 삭제는 문서 소유자만 가능) |

**안내:** 위키 블록은 참조만 담고 있으므로 삭제된 업무의 복구는 플로우에서 직접 수행합니다. 권한 복구도 플로우 프로젝트 관리자를 통해 별도로 요청합니다.

### 14-9. 신규 업무 생성 (Inline Create)

위키 에디터에서 바로 신규 플로우 업무를 생성할 수 있습니다.

| 단계 | 동작 |
|---|---|
| 진입 | `/플로우 업무` → "새 업무 만들기" 선택 |
| 프로젝트 선택 | 최근 사용 프로젝트 + 검색 (필수) |
| 제목 입력 | 인라인 입력 (필수) |
| 기본값 | 담당자 = 본인 · 상태 = 할 일 · 마감일 = 미설정 |
| 저장 후 | 생성된 업무 블록이 에디터에 삽입됨 + 플로우에 실시간 반영 |

### 14-12. 업무 상세 레이어 팝업 + 우측 AI 헬퍼 영역

리스트 뷰 업무 행 클릭 시 **플로우 게시물 팝업 스타일의 중앙 레이어 팝업**이 열리고, 동시에 **화면 우측 끝에 세로로 AI 헬퍼 영역**이 고정 노출됩니다.

**진입 방법:**

| 진입점 | 동작 |
|---|---|
| 블록 내 업무 행 클릭 | 팝업 + 우측 헬퍼 영역 동시 오픈 |
| 블록 내 ··· → "상세 보기" | 동일 |
| 키보드 Space (블록 선택 상태) | 동일 |

**팝업 헤더 우측 버튼 구성:**

| 버튼 | 동작 |
|---|---|
| **↗ 플로우로 가기** | 새 탭으로 플로우 업무 상세 페이지 이동 |
| ⎯ 최소화 | 팝업을 하단 작업바로 내림 |
| ✕ 닫기 | 팝업 + 헬퍼 영역 함께 닫힘 |

**팝업 구성 (플로우 게시물 스타일):**

| 영역 | 내용 |
|---|---|
| 헤더 | "업무 상세" 타이틀 · 프로젝트 라벨 · 우측 버튼 3종 |
| 제목 | 업무명 (크고 굵게) |
| 메타 정보 | 상태 · 마감일(D-N) · 담당자(+ 담당자 변경) · 진척도 · [+ 추가 항목 보기] |
| 본문 | 업무 설명 (자유 기재) |
| 하위업무 | 체크리스트 · [+ 추가] · [✨ AI 생성] |

**우측 AI 헬퍼 영역 (이번 단계는 영역만 확보):**

| 항목 | 값 |
|---|---|
| 위치 | **화면 우측 끝에 세로로 고정** (팝업과 분리된 독립 영역) |
| 폭 | 380px |
| 배경 | 흰색 (어둡힘 오버레이 영향 받지 않음) |
| 경계 | 좌측에 1px 구분선 |
| 내용 | **이번 버전은 영역만 확보 · 내부 UI는 추후 기획 예정** |

**레이아웃:**

| 항목 | 값 |
|---|---|
| 팝업 폭 / 높이 | 680px / auto (max 90vh) |
| 팝업 위치 | 화면 중앙 (좌측 0부터 우측 헬퍼 영역 직전까지의 영역 중앙) |
| 헬퍼 영역 폭 | 380px (화면 우측 고정) |
| 어둡힘 오버레이 | 화면 좌측 ~ 헬퍼 영역 직전까지 · 우측 헬퍼 영역은 오버레이 없음 |

| 항목 | 정책 |
|---|---|

### 14-13. 업무 삽입 모달 — 선택 리스트 (다중 프로젝트 가시화)

여러 프로젝트에 걸쳐 업무를 선택할 때 현재 담긴 업무를 확인할 수 있는 하단 리스트입니다.

**배경:** 현재 모달은 한 프로젝트 내 업무만 보여주고, 프로젝트를 전환하면 이전에 체크한 업무가 시각적으로 사라짐. 다중 프로젝트 선택 시 **무엇을 담았는지 파악 불가**.

**구성:**

| 영역 | 내용 |
|---|---|
| 상단 (기존 유지) | 프로젝트 드롭다운 · 검색 · 업무 테이블 |
| **하단 (신규)** | 선택된 업무 리스트 (프로젝트 전환과 무관하게 누적 유지) |

**선택 리스트 필드:**

| 컬럼 | 내용 |
|---|---|
| 체크박스 | 일괄 삭제용 (헤더 체크박스로 전체 선택/해제) |
| NO | 1부터 순차 번호 (선택 순서 기준) |
| 프로젝트명 | 해당 업무가 속한 프로젝트 |
| 업무명 | 업무 제목 (호버 시 전체 표시 툴팁) |
| 삭제 | 각 행의 ✕ 버튼 (개별 삭제) |

**동작:**

| 액션 | 결과 |
|---|---|
| 상단 테이블에서 체크 | 하단 선택 리스트에 자동 추가 |
| 상단 테이블에서 체크 해제 | 하단 리스트에서도 자동 제거 |
| 하단 리스트에서 ✕ 클릭 | 상단 테이블의 체크도 자동 해제 |
| 하단 체크박스 다중 선택 + [선택 삭제] | 일괄 삭제 |
| 프로젝트 전환 | 하단 리스트는 유지 (전환과 무관) |
| 빈 상태 | "아직 선택된 업무가 없어요" 플레이스홀더 |
| [삽입] 버튼 클릭 | 하단 리스트의 모든 업무를 각각 블록으로 에디터에 삽입 |

**UI 동작 디테일:**
- 선택 리스트는 최대 높이 180px, 초과 시 스크롤
- 선택된 업무 개수는 [삽입] 버튼에 표시: `삽입 (3)`
- 0개일 때 [삽입] 버튼 비활성

## 15. 데이터 마이그레이션 정책

> v2.4 신규 추가

### 15-1. 정책 개요

**목적:** 타 문서 도구(Notion · Confluence · Evernote · 로컬 Markdown 파일 등)에서 Flow Wiki로 기존 문서를 벌크 이관할 수 있도록 합니다.

**핵심 원칙:**
- **지원 포맷 최소주의**: `.md`, `.csv`만 지원. 이미지·PDF 등 첨부파일은 이번 단계 미지원 (이관 후 위키에서 직접 업로드).
- **구조 보존**: ZIP 업로드 시 파일 내부 폴더 구조를 그대로 재현.
- **투명성 우선**: 지원 외 파일이 있을 경우 스킵 목록을 사용자에게 명시 후 승인받음 (하이브리드 방식).
- **부분 성공 허용**: 일부 파일 실패 시에도 성공한 파일은 보존하고 실패 리포트 제공.

### 15-2. 진입점

마이그레이션은 **워크스페이스 설정 > 가져오기** 메뉴에서만 접근 가능합니다.

| 진입점 | 조건 |
|---|---|
| 설정 > 가져오기 | 최고 관리자 · 멤버 모두 접근 가능 |
| 일반 사이드바 · 에디터 | 노출 안 함 (대량 이관 전용 작업이므로) |

**사유:** 마이그레이션은 대량 작업이고 실행 후 되돌리기 어려우므로, 일반 문서 작성 플로우와 분리해 신중하게 접근하도록 유도.

### 15-3. 지원 포맷 및 제한

| 포맷 | 용도 | 변환 결과 |
|---|---|---|
| `.md` | Markdown 문서 | Wiki 문서 1개로 변환 (Markdown 파서 적용) |
| `.csv` | 스프레드시트 | Wiki 문서 1개 + 표 블록으로 변환 (첫 행 = 헤더) |
| `.zip` | 여러 파일 번들 | 압축 해제 후 내부 `.md` · `.csv`만 업로드 |

**미지원 포맷:** `.docx` · `.pdf` · `.html` · 이미지 파일 (`.png` · `.jpg` 등) · 음성/영상 파일. 프리뷰 모달에서 스킵 처리.

**용량 제한:**

| 항목 | 제한 |
|---|---|
| 단일 파일 최대 크기 | 10MB (`.md`), 20MB (`.csv`) |
| ZIP 파일 최대 크기 | 500MB |
| ZIP 내부 파일 개수 | 최대 1,000개 |
| ZIP 내부 폴더 깊이 | 최대 10단계 |
| 동시 업로드 세션 | 1개 (진행 중 추가 업로드 불가) |

### 15-4. 업로드 방식

**① 단일/다중 파일 업로드:**
- 드래그 & 드롭 또는 파일 선택
- 지정 위치: 사용자가 **대상 폴더 선택** (기본값: 내 문서)
- 파일 1개당 1개 문서로 변환

**② ZIP 업로드 (구조 보존):**

| 단계 | 동작 |
|---|---|
| 1. ZIP 업로드 | 드래그 & 드롭 또는 파일 선택 |
| 2. 압축 해제 | 서버에서 내부 구조 분석 |
| 3. 루트 폴더 생성 | **ZIP 파일명 = 루트 폴더명** (대상 위치 하위에 생성) |
| 4. 하위 폴더 생성 | ZIP 내부 폴더 구조 그대로 재현 (깊이 10단계까지) |
| 5. 문서 생성 | 각 `.md` · `.csv` 파일 → 해당 폴더 내 문서로 변환 |

**예시:**
```
my_notes.zip
├── meeting/
│   ├── 2026-04-01.md
│   └── 2026-04-08.md
└── plans.csv
```
↓ 업로드 후
```
📁 my_notes (ZIP 파일명으로 생성된 루트 폴더)
├── 📁 meeting
│   ├── 📄 2026-04-01
│   └── 📄 2026-04-08
└── 📄 plans
```

### 15-5. 프리뷰 모달 (하이브리드 승인)

ZIP 업로드 시 압축 해제 후 내부 파일을 분석하여 사용자에게 프리뷰를 보여주고, 명시적 승인을 받은 후에만 실제 이관을 실행합니다.

**프리뷰 구성:**

| 영역 | 내용 |
|---|---|
| 업로드 요약 | ZIP 파일명 · 총 파일 수 · 폴더 구조 트리 |
| **업로드 대상** | 변환 가능한 `.md` · `.csv` 파일 수 (초록) |
| **스킵 대상** | 지원 외 파일 목록 (주황) — 파일명 + 확장자 노출 |
| 에러 대상 | 손상된 파일 · 크기 초과 파일 (빨강) |

**사용자 선택:**

| 버튼 | 동작 |
|---|---|
| [취소] | 업로드 중단 · ZIP 파기 |
| [스킵하고 업로드] | 지원 외 파일 제외 + 나머지 실행 |
| [취소하고 ZIP 다시 정리] | 사용자가 파일 정리 후 재업로드 |

**예외 케이스:**
- 변환 가능한 파일이 **0개**인 경우 → [업로드] 버튼 비활성 + "변환 가능한 파일이 없어요" 안내
- 전부 지원 외 파일 → 프리뷰 모달에서 에러 메시지와 함께 취소만 가능

### 15-6. 파일명 처리

**파일명 정책:** 원본 파일명 유지를 원칙으로 합니다.

| 케이스 | 처리 |
|---|---|
| 한글·영문·숫자 | 그대로 유지 |
| 특수문자 (`!@#$%` 등) | 그대로 유지 |
| OS 금지 문자 (`/` `\` `:` `*` `?` `"` `<` `>` `|`) | 언더스코어(`_`)로 치환 |
| 확장자 제거 | `.md` · `.csv` 확장자는 제외하고 문서명으로 저장 |
| 동일 폴더 내 중복 이름 | 뒤에 `(2)`, `(3)` 등 자동 부여 |
| 파일명 공백 앞뒤 | 자동 trim |
| 빈 파일명 | "제목 없음" 으로 저장 |

**인코딩:** UTF-8 기준 유니코드 파일명 처리. Windows/Mac/Linux에서 생성된 zip 모두 지원.

### 15-7. 변환 규칙

**Markdown (`.md`) → Wiki 문서:**

| Markdown 요소 | Wiki 블록 |
|---|---|
| `# H1` ~ `###### H6` | 제목 블록 (H1/H2/H3까지만 · H4+ 는 H3로 내림) |
| `**bold**` · `*italic*` · `~~strike~~` | 인라인 서식 그대로 |
| `- item` / `1. item` | 불릿/순서 리스트 |
| `- [ ] todo` | 체크박스 블록 |
| `> quote` | 인용 블록 |
| ` ```code``` ` | 코드 블록 (언어 감지) |
| `[link](url)` | 링크 |
| `![image](url)` | **이미지 URL만 유지 (프리뷰 미노출)** · 이관 후 수동 재업로드 필요 |
| `| table |` | 표 블록 |

**CSV (`.csv`) → Wiki 문서:**
- 첫 행 = 헤더로 인식 (첫 행 이름으로 컬럼 구성)
- 최대 50컬럼 · 1,000행 (초과 시 분할 문서 생성)
- 빈 컬럼/행 자동 정리

### 15-8. 실패 처리 및 부분 성공

마이그레이션 중 일부 파일 실패 시에도 **성공한 파일은 보존**합니다.

**실패 케이스:**

| 케이스 | 처리 |
|---|---|
| 파일 손상 | 해당 파일 스킵 + 리포트 기록 |
| 인코딩 오류 | UTF-8이 아닐 시 자동 감지 시도 → 실패 시 스킵 |
| 크기 초과 | 업로드 전 차단 (프리뷰에서 제외) |
| 서버 오류 | 3회 재시도 후 실패 처리 |
| 네트워크 끊김 | 중단 지점부터 재개 가능 (세션 30분 유지) |

**실패 리포트:**

| 항목 | 내용 |
|---|---|
| 성공 | 성공한 파일 개수 + 생성된 문서 링크 |
| 실패 | 실패 파일명 + 실패 사유 + 재시도 버튼 |
| 스킵 | 사용자가 프리뷰에서 승인한 스킵 목록 |
| 다운로드 | 리포트를 `.csv`로 다운로드 가능 |

### 15-9. 이관 이력 관리

"완료됨" 탭에서 과거 마이그레이션 이력을 관리합니다.

| 항목 | 내용 |
|---|---|
| 이관 일시 | 업로드 시작 시점 |
| 업로드 파일 | ZIP 파일명 또는 파일 개수 |
| 대상 폴더 | 업로드된 위치 경로 |
| 결과 | 성공 N / 스킵 N / 실패 N |
| 액션 | 리포트 보기 · 실패 항목 재시도 · 업로드된 문서 열기 |

**이력 보관 기간:** 90일 (90일 경과 시 자동 삭제, 업로드된 문서 자체는 유지)

### 15-10. 권한 및 제한

| 역할 | 권한 |
|---|---|
| 최고 관리자 | 모든 스코프(회사 도서관·공유 서재·내 문서)로 이관 가능 |
| 멤버 | 내 문서 · 본인이 속한 공유 서재로만 이관 가능 |

**속도 제한:**
- 사용자당 **일 10회** 업로드 세션 제한 (대량 업로드 방지)
- 동일 워크스페이스 동시 업로드 5개 제한

### 15-11. 담당 범위

## 16. 어드민 페이지 정책 (MVP)

### 16-1. 정책 개요

**목적:** Flow 위키 어드민 페이지는 최고관리자가 **회사 도서관 · 멤버 · 보안 · 요금제**를 관리할 수 있는 전용 공간입니다.

| 권한 없음 | 패널 + 토글 버튼 모두 비노출. §16 요금제 안내로 유도. |

### 18-6. 사용 한도 초과 시 정책

| 원칙 | 처리 |
|---|---|
| 패널 가시성 | 숨기지 않음 (사용자가 기능 존재를 인지하도록) |
| 입력창 | 비활성화 + 안내 문구 노출 |
| 이전 생성 결과 | 계속 활용 가능 ([현재 위치에 추가] / [복사] 활성화 유지) |
| 한도 리셋 | 리셋 후 재진입 시 자동으로 정상 상태 복귀 |

### 18-7. 진입 시 패널 상태 결정

| 조건 | 동작 |
|---|---|
| 이전 세션에서 패널 열림으로 종료 | 패널 열림 상태로 진입 |
| 이전 세션에서 패널 닫힘으로 종료 | 패널 닫힘 상태로 진입 (토글 버튼만 노출) |
| 최초 진입 (이전 기록 없음) | **패널 열림** 상태로 진입 (기능 인지 목적) |
| AI 헬퍼 사용 권한 없음 | 패널 + 토글 버튼 모두 비노출 |

> 패널 너비 / 열림 상태는 사용자별 로컬 저장. 다음 진입 시 복원.

### 18-8. 추가 위치 결정 규칙 (커서 상태별)

| 좁은 화면 (1120px 미만) | 헬퍼 패널 자동 숨김 · 토글 버튼만 노출 |
| 모바일 (768px 미만) | 우측 패널 → 하단 시트(Bottom Sheet)로 전환 · 토글 버튼은 우하단 floating |

### 18-12. 유의사항

- AI 헬퍼 패널은 **편집 모드에서만** 노출 (보기 모드에서는 노출 X)
- 동시 편집 중인 타 사용자에게는 본인의 AI 헬퍼 패널이 노출되지 않음 (각자 본인 화면에서만 사용)
- 추가된 블록은 **단일 Undo (Ctrl+Z)** 로 일괄 제거 가능해야 함
- 어드민 페이지 §16에서 워크스페이스 단위로 AI 헬퍼 활성화/비활성화 제어 가능
- 한도 초과 안내 문구 톤은 Flow AI 요금 및 운영 정책 문서와 일치

## 19. 버전 히스토리 정책

> v4.5 신규 추가 — 화면 정의서 p17 기반

### 19-1. 정책 개요

문서 편집 시 자동으로 누적되는 버전 기록을 조회·미리보기·비교·복원할 수 있는 기능. 협업 환경에서 변경 추적성을 보장한다.

| 항목 | 내용 |
|---|---|
| 진입 경로 | 에디터 상단바 "🕐 히스토리" 버튼 |
| 모달 형태 | 좌우 분할 (좌측 버전 리스트 / 우측 본문 미리보기) |
| 보관 기간 | 최근 30일 |
| 권한 | 문서 조회 권한 이상 (편집자·소유자·최고 관리자) |

### 19-2. 버전 자동 기록 조건

| 트리거 | 처리 |
|---|---|
| 문서 저장 (자동저장 포함) | 변경 사항 있을 시 신규 버전 생성 |
| 동시 편집 종료 | 마지막 편집자 기준으로 1버전 통합 기록 |
| 5분 이내 동일 사용자 연속 편집 | 1버전으로 묶어 기록 (debounce) |
| 사용자 명시적 저장(Ctrl+S) | 즉시 신규 버전 생성 |

### 19-3. 버전 리스트 구성

좌측 패널 — 시간순 그룹핑 (오늘 / 어제 / 날짜별).

| 항목 | 정의 |
|---|---|
| 시간 | HH:MM 형식 (당일은 시간만, 이전은 날짜+시간) |
| 편집자 | 프로필 아바타 + 이름·직책 |
| 변경 요약 | "+ N줄 수정" / "+ N줄 · - N줄" / "표 추가" / "이미지 추가" / "제목 변경" / "+ N줄 · 섹션 재구성" |
| 현재 버전 표시 | 최상단에 "현재" 라벨 강조 |

### 19-4. 상태별 처리 정책

| 상태 | 동작 |
|---|---|
| DEFAULT (모달 진입) | 최신 버전(현재) 자동 선택. 우측에 현재 본문 노출. |
| PREVIEW (이전 버전 클릭) | 우측에 해당 버전 읽기 전용 미리보기. "읽기 전용" 라벨 노출. |
| COMPARE (비교 모드) | 우측에 diff 하이라이트 (추가 = 초록 / 삭제 = 빨강). 비교 대상 버전명 상단 표시. |
| RESTORE_CONFIRM (복원 클릭) | 확인 다이얼로그 — "이 버전으로 복원하시겠어요? 현재 내용은 새 버전으로 자동 저장됩니다." |

### 19-5. 액션 버튼 정의

| 버튼 | 위치 | 동작 |
|---|---|---|
| 이 버전과 비교 | 우측 상단 (PREVIEW 상태) | COMPARE 모드 진입 |
| 이 버전으로 복원 | 우측 상단 (PREVIEW 상태) | RESTORE_CONFIRM 진입 |
| 비교 종료 | 우측 상단 (COMPARE 상태) | PREVIEW 모드로 복귀 |
| 닫기(×) | 모달 우상단 | 모달 종료 |

### 19-6. 복원 처리 정책

| 단계 | 처리 |
|---|---|
| 복원 요청 | 사용자가 [이 버전으로 복원] 클릭 |
| 안전장치 | 복원 직전 현재 본문을 자동으로 신규 버전으로 저장 (변경 손실 방지) |
| 복원 실행 | 선택한 버전 내용을 현재 본문으로 덮어쓰기 |
| 신규 버전 기록 | "N월 N일 N시 버전으로 복원" 메모와 함께 신규 버전 생성 |
| 알림 | 동시 편집 중인 타 멤버에게 "N님이 이전 버전으로 복원했습니다" 토스트 노출 |

### 19-7. 권한 정책

| 행위 | 가능한 사람 |
|---|---|
| 버전 히스토리 조회 | 조회자 + 편집자 + 소유자 + 최고 관리자 |
| 버전 미리보기 | 조회자 + 편집자 + 소유자 + 최고 관리자 |
| 비교 모드 | 조회자 + 편집자 + 소유자 + 최고 관리자 |
| 이전 버전으로 복원 | 편집자 + 소유자 + 최고 관리자 (조회자 불가) |

### 19-8. 권한 변경 시 버전 히스토리 처리

| 상황 | 처리 |
|---|---|
| 편집 권한이 조회로 변경 | 기존 버전 기록 유지. 새 버전 생성 권한만 차단. |
| 권한 회수 | 본인이 만든 버전도 조회 불가 (문서 자체 비노출) |
| 소유자 이전 | 모든 버전 기록 신규 소유자에게 승계 |

### 19-9. 보관 기간 및 자동 삭제

| 항목 | 정책 |
|---|---|
| 기본 보관 | 최근 30일 |
| 30일 경과 버전 | 자동 영구 삭제 (복구 불가) |
| 최소 보관 보장 | 30일 이내라도 최소 5개 버전은 항상 보존 |
| 영구 보존 마킹 | 향후 지원 예정 (v2.0) |

### 19-10. 예외 처리

| 상황 | 처리 |
|---|---|
| 문서 본문이 매우 큰 경우 (10MB 이상) | 미리보기는 일부만 로드 + "더 보기" |
| diff 계산 실패 | "비교를 표시할 수 없습니다. 미리보기로 확인하세요." |
| 복원 중 네트워크 단절 | 복원 미실행 + "다시 시도" 안내. 자동 저장된 안전 버전은 유지. |
| 30일 경계 시점 버전 | 삭제 직전 안내 없음 (사용자에게 별도 알림 X) |
| 동시 편집 중 복원 시도 | "현재 N명이 편집 중입니다. 편집 종료 후 다시 시도해주세요." 안내 후 차단 |
